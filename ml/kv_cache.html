Section 00 · Why this page exists

Training taught you attention.
Serving teaches you memory.

During training, we talk about expressive power, optimization, and attention patterns. During deployment, the bottlenecks shift. The model weights sit there; the requests move; and the thing that keeps growing token by token is the KV cache.

This page is about that second world. Not "what is a transformer?" but "why is time-to-first-token high?", "why does batch size fall as the conversation gets longer?", "why does GQA matter so much for serving?", and "what exactly does vLLM mean by PagedAttention?"

Section 01 · Mental model

Serving a causal LM — one new query, old keys and values.

At generation step t, a decoder-only transformer does not recompute attention for the whole sequence from scratch. It computes a new query for the newest token, compares that query against all previously stored keys, mixes the stored values, and appends one more key/value pair to memory.

That change of perspective is the entire point of KV caching. During training we materialize many query rows at once. During inference, after the prompt has been prefetched, the hot loop is narrower: one token arrives, one token leaves, and the history is reused.

Without cache

Every decode step would rerun projections and attention over the entire prefix seen so far. Correct, but wasteful: the model would keep recomputing identical old keys and values.

With cache

Old K and V tensors are stored once per layer. Each new step computes only the fresh row, attends over the stored history, then appends the row to the cache.

The serving mindset

Think of autoregressive inference as a stateful program. The model weights are immutable. The request state is the prompt prefix plus the per-layer KV cache. Most serving tricks are really different ways of managing that state efficiently.

Section 02 · Reuse

KV caching — save the history, append the frontier.

The cache stores the per-layer key and value tensors for all tokens seen so far. On the next token, the model computes a single new row of K and V, concatenates them to the cache, and reuses the rest.

The decode step, written explicitly

For the newest token, attention at one layer reduces to a single query row looking back over the cached history:

\[ o_t = \operatorname{softmax}\!\left(\frac{q_t K_{1:t-1}^{\top}}{\sqrt d}\right) V_{1:t-1} \] Eq. 1 · one-token decode with a cache

The expensive part is not the math on paper. It is the fact that for every new token, every layer must read a longer and longer cache from high-bandwidth memory. This is why long-context serving often turns into a memory-traffic problem long before it becomes a raw FLOP problem.

decode_step.pypedagogical one-layer cache append 
def decode_step(x_t, cache_k, cache_v, W_q, W_k, W_v):
    # x_t: (B, d_model) newest token embedding
    # cache_k, cache_v: (B, T_prev, d_head) for one head in one layer

    q_t = W_q(x_t)[:, None, :]                          # (B, 1, d_head)
    k_t = W_k(x_t)[:, None, :]
    v_t = W_v(x_t)[:, None, :]

    K = torch.cat([cache_k, k_t], dim=1)              # append to history
    V = torch.cat([cache_v, v_t], dim=1)

    scores = q_t @ K.transpose(-2, -1) / math.sqrt(q_t.size(-1))
    probs  = F.softmax(scores, dim=-1)
    out_t  = probs @ V
    return out_t, K, V

The weights are fixed once the model is loaded. What grows with request length is the cache of old keys and values that every future decode step must read.

How many bytes are we talking about?

One cached token stores both keys and values, for every layer, for every KV head. A good back-of-the-envelope formula is:

\[ \text{KV bytes} \approx B \cdot L \cdot T \cdot 2 \cdot H_{\mathrm{kv}} \cdot d_{\mathrm{head}} \cdot s \] Eq. 2 · cache size for a batch of active sequences

Here B is active batch size, L is layer count, T is tokens already cached, Hkv is the number of key/value heads, and s is bytes per element. The factor of 2 is because you store both K and V.

Section 03 · Throughput shift

Prefill vs decode — the bottleneck flips after the prompt.

Prefill consumes the input prompt and builds the cache. Decode uses that cache to emit output tokens one by one. The first phase is a big, parallel matrix workload. The second is a skinny, serial loop with ever larger memory reads.

This is why time-to-first-token and tokens-per-second are often governed by different mechanisms. Prefill likes large GEMMs and high arithmetic intensity. Decode likes low-overhead scheduling and careful cache management.

\[ \text{work without cache} \;\approx\; \sum_{t=1}^{G} O\!\bigl((P+t)^2\bigr), \qquad \text{work with cache} \;\approx\; O(P^2) + \sum_{t=1}^{G} O(P+t) \] Eq. 3 · why caching changes decode from repeated quadratic recompute to incremental reuse

A common mistake

People often say "inference is just a forward pass." True, but misleading. Serving performance is not about one forward pass in isolation. It is about a long-lived stateful process that alternates a prompt build-up phase with thousands or millions of one-token updates.

Section 04 · Head economics

GQA and MQA — fewer KV heads, smaller serving bill.

During training, grouped-query attention may sound like a representational design choice. During serving, it is also a direct memory optimization. The cache scales with KV heads, not with the number of query heads.

MHA

Every query head gets its own key and value head. Best symmetry, worst cache footprint.

GQA

Several query heads share one KV head group. This is the modern compromise in many deployed LLMs.

MQA

All query heads share one key head and one value head. Minimum KV memory, but the strongest sharing assumption.

If a model has 32 query heads and only 8 KV heads, the cache is roughly one quarter of the MHA baseline, everything else equal. That reduction flows through the whole serving stack: more active requests fit, larger contexts fit, and bandwidth pressure during decode falls.

gqa_shapes.pythe cache follows KV heads 
def kv_cache_shape(batch, layers, tokens, kv_heads, head_dim):
    # one cache tensor for keys, one for values
    K = (batch, layers, tokens, kv_heads, head_dim)
    V = (batch, layers, tokens, kv_heads, head_dim)
    return K, V

Section 05 · Memory manager

PagedAttention — virtual memory for the KV cache.

A naive engine tries to store each request's cache in one contiguous chunk. That is simple, but it fragments badly as requests of different lengths arrive and finish. PagedAttention breaks each request into fixed size blocks and keeps a small block table that maps logical blocks to physical ones.

Logical tokens, physical blocks

The idea is operating-systems flavored. A request thinks in logical token positions. The allocator thinks in uniform cache pages. The block table bridges the two: "logical block 0 lives in physical block 12, logical block 1 lives in physical block 4," and so on.

Why this beats contiguous buffers

With contiguous allocation, a new request may fail to fit even when the total free memory is sufficient, simply because the free space is scattered. With fixed-size blocks, scattered free blocks are still useful. The allocator cares about how many blocks are free, not whether they form one uninterrupted run.

The attention kernel still has to follow the mapping

PagedAttention is not just an allocator. The attention kernel must read the block table, walk the physical blocks in the right order, and gather the right key/value tiles for each sequence in the batch. The logical sequence looks contiguous to the model, even though the memory is not.

paged_attention_sketch.pyconceptual block-table traversal 
def paged_attention_step(q_t, K_pages, V_pages, block_table, seqlen, block_size):
    logical_blocks = math.ceil(seqlen / block_size)
    phys = block_table[:logical_blocks]                         # map logical -> physical

    K_parts, V_parts = [], []
    for b in phys:
        K_parts.append(K_pages[b])
        V_parts.append(V_pages[b])

    K = torch.cat(K_parts, dim=0)[:seqlen]
    V = torch.cat(V_parts, dim=0)[:seqlen]

    scores = q_t @ K.transpose(-2, -1)
    probs = F.softmax(scores, dim=-1)
    return probs @ V
The key systems win

Paged allocation makes it far easier to support continuous batching, because requests can grow incrementally without demanding that the allocator reserve one giant contiguous region up front.

Section 06 · Scheduler

Continuous batching — schedule at the token step, not at the request.

Static batching waits for a whole group of requests to finish before the next group enters. Continuous batching, also called iteration-level scheduling, admits and retires requests every decode step. That sounds like a small change. In practice it is a huge utilization change.

Generation lengths are variable. One request may stop after 12 tokens, another after 300. If you force them to stay together until the longest one finishes, GPU slots go idle. Continuous batching fills those holes immediately.

scheduler_loop.pyiteration-level serving loop 
while waiting_queue or active_batch:
    # retire completed sequences
    active_batch = [req for req in active_batch if not req.done]

    # admit new work into any freed slot
    while len(active_batch) < max_slots and waiting_queue:
        active_batch.append(waiting_queue.pop(0))

    # run exactly one decode iteration for everyone active
    logits = model_decode_one_token(active_batch)
    sample_and_append(active_batch, logits)
Scheduler realism

Once you allow sequences to enter and leave every iteration, batching, allocation, and stop conditions become tightly coupled. This is why serving engines look more like operating systems than like notebook inference scripts.

Section 07 · Reuse across requests

Prefix caching — if the prompt prefix is identical, do not pay twice.

Many workloads repeat the same system prompt, the same long document header, or the same few-shot examples across many requests. Prefix caching reuses the already-built KV cache for that shared prefix instead of rebuilding it every time.

The important word is identical. This is not fuzzy semantic matching. It is exact token-prefix reuse. If two requests share the first 1200 tokens byte-for-byte after tokenization, the second one can skip recomputing those 1200 tokens and begin from the cached state.

prefix_cache.pyexact prefix reuse 
def lookup_prefix_cache(token_ids, block_hash_table):
    matched_blocks = []
    parent = None
    for block_tokens in chunk(token_ids, block_size):
        h = hash((parent, tuple(block_tokens)))
        if h not in block_hash_table:
            break
        matched_blocks.append(block_hash_table[h])
        parent = h
    return matched_blocks
Operational subtlety

Prefix caching helps only when prefix equality happens on the tokenized input seen by the model. A tiny difference in whitespace, tool schema order, or hidden templating can destroy cache hits.

Section 08 · Long context

Long-context serving tricks — stretch the cache without drowning in it.

KV caching is necessary, but not always sufficient. As context grows, modern engines layer on extra tricks: windowed attention, chunked prefill, and KV quantization. Each one targets a different pain point.

Sliding window

Keep only the recent tail that the model actually attends to. Best when architecture and task both tolerate locality.

Chunked prefill

Break enormous prompts into pieces so prompt ingestion can coexist with decode traffic instead of monopolizing the GPU.

KV quantization

Store cached keys and values in fewer bits to reduce memory footprint and bandwidth, at some accuracy risk.

sliding_window_cache.pyring-buffer style tail retention 
def append_windowed(cache_k, cache_v, k_t, v_t, window):
    cache_k = torch.cat([cache_k, k_t], dim=1)[..., -window:, :]
    cache_v = torch.cat([cache_v, v_t], dim=1)[..., -window:, :]
    return cache_k, cache_v
Different tricks solve different problems

PagedAttention solves allocation waste. Continuous batching solves scheduler waste. Prefix caching solves repeated prefill waste. Sliding windows and KV quantization solve raw cache size. They complement each other; they are not substitutes.

Section 09 · End-to-end picture

A modern serving stack — allocator, scheduler, cache reuse, and architecture.

Real deployments rarely rely on one trick. High-throughput engines stack them: smaller KV heads at the model level, paged allocation at the memory level, continuous batching at the scheduler level, and prefix reuse when workloads repeat themselves.

Technique What It Fixes Best When Main Cost / Caveat
KV caching Avoids recomputing old keys and values every step Any autoregressive decode workload Cache grows with context and must be read every token
GQA / MQA Shrinks per-token cache bytes and decode bandwidth Long contexts, high concurrency Architectural tradeoff; not free to retrofit after training
PagedAttention Reduces fragmentation and enables flexible growth Variable-length requests under continuous batching Allocator and kernel become more complex
Continuous batching Improves GPU utilization under variable output lengths Online serving with heterogeneous requests Scheduler, stop logic, and allocation interact tightly
Prefix caching Skips repeated prefill on identical prompt prefixes Shared system prompts, repeated document scaffolds Exact token prefix match required
Sliding window / KV quantization Reduces absolute cache footprint for long contexts Architectures or workloads tolerant to approximation Potential quality loss or architectural constraints

Questions to ask when tokens/sec is disappointing

Are you decode-bound or prefill-bound? Is the cache too large because of MHA? Are requests fragmenting memory? Are short generations sitting idle behind one long response? Are repeated prompts missing cache hits because the prompt template is not stable? Most production issues land in one of those buckets.

One-line summary

Attention explains what the model means. KV caching, paged attention, and scheduling explain what the model costs.
