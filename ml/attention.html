Section 00 · Motivation

Why attention ate deep learning.

Before 2017, sequence models were recurrent. Information flowed through time one step at a time, gradients vanished, and a token at position 1000 had a feeble grip on a token at position 1. Attention rewrote the rules: every position can talk to every other position in a single matrix multiply, and the network learns who should talk to whom.

This document builds the mechanism from the ground up. We start with a single dot product between two vectors, scale it, soften it, stack it into multiple heads, give it positional awareness, make it cheaper, make it longer, make it specialised — and end with a small transformer you can train in a notebook.

Throughout, we use a consistent colour code: queries are amber, keys are teal, values are rose, and the output of attention is violet. If something is amber in one figure, it is amber in every figure.

What you'll build

Scaled dot-product attention, multi-head attention, sinusoidal + RoPE + ALiBi positional encodings, sliding-window and block-sparse masks, ProbSparse, linear attention, Gated DeltaNet, FlashAttention sketch, GQA/MQA/MLA, cross-attention, time-series-specific temporal and spectral attention, and a 12-line training loop.

What you'll understand

Why \( \sqrt d \) appears. Why softmax. Why heads. Why positional info has to be added in. Why attention is \( O(N^2) \), and the several different ways people try to escape that. Why FlashAttention won despite doing more arithmetic.

What you'll see

Twelve interactive widgets — attention as soft retrieval, scaling effects, MHA decomposition, sinusoidal heatmaps, RoPE rotations, mask gallery, complexity curves, linear-attention associativity, tiled attention, GQA vs MQA, seasonal-lag discovery, and a live attention heatmap explorer.

Section 01 · Foundations

Basic attention — soft retrieval.

The single mental model that unlocks everything: attention is a differentiable hash table. A query compares itself to every key, the resulting similarities become weights via softmax, and those weights select a mixture of values.

From hash table to attention

In a regular Python dict, you look up d[q] by finding the key that exactly matches q and returning its associated value. There is one winner; the rest are ignored. Attention is the soft, differentiable version of this: the lookup is a weighted average over all values, with weights determined by how similar each key is to the query.

Make this concrete. Say we have N=5 tokens, each represented by a key vector and a value vector. We arrive with one query. We compute a similarity score with each of the 5 keys (one number per key), normalise the 5 numbers into a probability distribution with softmax, and return the corresponding convex combination of the 5 value vectors. That's it.

\[ \operatorname{attention}(q, K, V) = \operatorname{softmax}\!\left(\frac{qK^\top}{\sqrt d}\right)V \] Eq. 1 · scaled dot-product

Why dot product?

Many similarity functions would work — cosine, negative Euclidean, MLPs (additive attention, used in early NMT). Dot product wins for two reasons. Speed: it parallelises into a single matrix multiplication, which is the operation GPUs eat for breakfast. Flexibility: the magnitude of the dot product, not just its angle, carries information — a high-magnitude query can dominate a softmax without changing its direction.

Why scaled?

If q and k are independent random vectors with components of unit variance and dimension d, then their dot product has variance d. As d grows, scores spread out, and softmax becomes peaked — almost a hard argmax. Gradients through that peak are tiny. Dividing by \( \sqrt d \) normalises the variance back to 1 and keeps softmax in its useful regime.

Q, K, V from the same source: self-attention

In self-attention, Q, K, and V all come from the same input X via three separate linear projections:

attention.pyscaled dot-product self-attention 
import torch
import torch.nn.functional as F
import math


def scaled_dot_product_attention(Q, K, V, mask=None):
    """
    Q, K, V: (..., seq_len, d_k)
    mask:     (..., seq_len_q, seq_len_k) with 0 = forbidden, 1 = allowed
    Returns:  output (..., seq_len_q, d_k), weights (..., seq_len_q, seq_len_k)
    """
    d_k = Q.size(-1)

    # 1. raw similarity scores
    scores = Q @ K.transpose(-2, -1) / math.sqrt(d_k)

    # 2. apply mask (e.g. causal, padding)
    if mask is not None:
        scores = scores.masked_fill(mask == 0, float('-inf'))

    # 3. normalise to a distribution over keys
    weights = F.softmax(scores, dim=-1)

    # 4. weighted sum of values
    output = weights @ V
    return output, weights

Four lines of arithmetic. That's the entire attention operation. Everything else in this document is a wrapper, a variant, or a way to make those four lines faster.

Section 02 · Heads

Multi-head attention — many lenses.

A single attention head learns one type of relationship. Multi-head attention runs H heads in parallel, each on a smaller subspace, then concatenates and re-projects. One head might track syntactic agreement, another lexical co-occurrence, another long-range dependency. The model gets to learn all of them at once, cheaply.

The arithmetic of splitting

We do not run H independent attention computations on d-dimensional vectors — that would multiply the cost. Instead, we project X to a d-dimensional Q/K/V once, then reshape each into H heads of dimension d/H. Each head operates on its slice in parallel.

For d = 512 and H = 8, each head sees vectors of dimension 64. Eight 64-dim attentions cost the same as one 512-dim attention, because matmul cost is linear in feature dim once the sequence-length term dominates. We get diversity at the same FLOP budget.

mha.pyfull manual implementation 
import torch.nn as nn


class MultiHeadAttention(nn.Module):
    def __init__(self, d_model: int, num_heads: int, dropout: float = 0.0):
        super().__init__()
        assert d_model % num_heads == 0, "d_model must be divisible by num_heads"
        self.d_model   = d_model
        self.num_heads = num_heads
        self.d_k       = d_model // num_heads

        # Single matrix per role — heads are a reshape, not separate parameters
        self.W_q = nn.Linear(d_model, d_model, bias=False)
        self.W_k = nn.Linear(d_model, d_model, bias=False)
        self.W_v = nn.Linear(d_model, d_model, bias=False)
        self.W_o = nn.Linear(d_model, d_model, bias=False)
        self.drop = nn.Dropout(dropout)

    def forward(self, x, mask=None):
        B, N, _ = x.shape

        # project then split into heads: (B, N, d) -> (B, H, N, d_k)
        Q = self.W_q(x).view(B, N, self.num_heads, self.d_k).transpose(1, 2)
        K = self.W_k(x).view(B, N, self.num_heads, self.d_k).transpose(1, 2)
        V = self.W_v(x).view(B, N, self.num_heads, self.d_k).transpose(1, 2)

        # scaled dot-product, per head (broadcasts over B and H)
        scores = Q @ K.transpose(-2, -1) / math.sqrt(self.d_k)
        if mask is not None:
            scores = scores.masked_fill(mask == 0, float('-inf'))
        attn = self.drop(F.softmax(scores, dim=-1))
        out  = attn @ V                              # (B, H, N, d_k)

        # merge heads back: (B, H, N, d_k) -> (B, N, d)
        out = out.transpose(1, 2).contiguous().view(B, N, self.d_model)
        return self.W_o(out), attn

The shape gymnastics, slowly

The reshape .view(B, N, H, d_k).transpose(1, 2) is the only conceptually new operation here. After W_q(x), our tensor is (B, N, d_model). We re-interpret the last axis as (H heads × d_k features-per-head), giving (B, N, H, d_k). Transposing to (B, H, N, d_k) puts heads next to batch so PyTorch broadcasts attention across them. After attention, we transpose and view back, and finally apply W_o to mix information across heads.

What heads learn (empirically)

Probing studies on trained transformers consistently find heads that specialise. Some examples seen repeatedly across models:

Positional heads — attend mostly to the previous or next token. Syntactic heads — attend to syntactic dependents (subject ↔ verb). Copying heads — attend to identical tokens earlier in the sequence (useful for named entities). Diffuse heads — broadly attend everywhere; they act as a soft "summarise everything" channel. Most learned heads are some interpolation of these archetypes.

Implementation detail

We use .contiguous() after the transpose because view requires a contiguous memory layout, and transpose only updates the strides without moving data. Forgetting this will give you a confusing RuntimeError at the next view.

Section 03 · Position

Positional encoding — injecting order.

The attention operation we built has no notion of position. Permute the input tokens and the output permutes identically — useful for sets, fatal for sequences. We must inject positional information explicitly. The three dominant strategies are sinusoidal embeddings (the original), RoPE (the modern default), and ALiBi (the long-context favourite).

Sinusoidal positional embeddings

The original Transformer added a fixed (non-learned) positional vector to every input embedding. The dimensions alternate between sines and cosines at geometrically increasing wavelengths:

\[ \mathrm{PE}(pos, 2i)=\sin\!\left(\frac{pos}{10000^{2i/d}}\right), \qquad \mathrm{PE}(pos, 2i+1)=\cos\!\left(\frac{pos}{10000^{2i/d}}\right) \] Eq. 2 · sinusoidal PE

Why this exact form? Two reasons. First, every dimension is bounded in [−1, 1], so the magnitudes do not explode for distant positions. Second, for any fixed offset k, the encoding at position pos + k is a fixed linear transformation of the encoding at pos — a property that, in theory, lets the model recover relative offsets via linear combinations of absolute encodings.

sinusoidal_pe.py 
def sinusoidal_pe(seq_len: int, d_model: int) -> torch.Tensor:
    pe = torch.zeros(seq_len, d_model)
    pos = torch.arange(seq_len).unsqueeze(1).float()                     # (N, 1)
    div = torch.exp(torch.arange(0, d_model, 2).float()
                    * -(math.log(10000.0) / d_model))             # (d/2,)
    pe[:, 0::2] = torch.sin(pos * div)
    pe[:, 1::2] = torch.cos(pos * div)
    return pe                                                            # (N, d)

# Add to token embeddings — same shape, element-wise
x = token_embedding(input_ids) + sinusoidal_pe(N, d_model)

Rotary Positional Embedding (RoPE)

RoPE, introduced by Su et al. (2021), takes a different approach. Instead of adding a position vector to the embeddings, it rotates consecutive pairs of features in Q and K by an angle that depends on the absolute position. The magic is in the algebra: when you take a dot product between a rotated query at position m and a rotated key at position n, the result depends only on the relative position m − n.

\[ \langle R_m q,\; R_n k \rangle = \langle q,\; R_{n-m} k \rangle \] Eq. 3 · RoPE's key identity

Concretely, pair up the d features into d/2 pairs. The i-th pair gets rotated by angle m · θi, where θi = 10000^(−2i/d). Pairs with small i rotate fast (capture short-range patterns); pairs with large i rotate slowly (capture long-range structure). Same frequency schedule as sinusoidal — different application.

rope.pyHuggingFace-style implementation 
def precompute_rope(seq_len: int, d: int, base: float = 10000.0):
    """Precompute cos/sin tables for RoPE. d is the per-head dim."""
    inv_freq = 1.0 / (base ** (torch.arange(0, d, 2).float() / d))   # (d/2,)
    t = torch.arange(seq_len).float()                                          # (N,)
    freqs = torch.outer(t, inv_freq)                                            # (N, d/2)
    # duplicate so we have one angle per feature
    emb = torch.cat([freqs, freqs], dim=-1)                                  # (N, d)
    return emb.cos(), emb.sin()                                              # (N, d), (N, d)


def rotate_half(x):
    x1, x2 = x.chunk(2, dim=-1)
    return torch.cat([-x2, x1], dim=-1)


def apply_rope(q, k, cos, sin):
    """q, k: (B, H, N, d).  cos, sin: (N, d)."""
    cos = cos[None, None, :, :]              # broadcast over B, H
    sin = sin[None, None, :, :]
    q_rot = q * cos + rotate_half(q) * sin
    k_rot = k * cos + rotate_half(k) * sin
    return q_rot, k_rot

#   cos, sin = precompute_rope(N, d_k)

ALiBi — attention with linear biases

ALiBi (Press et al., 2022) is the simplest of the three. No embeddings, no rotations. Just add a fixed negative bias to attention scores proportional to the distance between query and key:

\[ \mathrm{scores}[i,j] \leftarrow \mathrm{scores}[i,j] - m\,|i-j| \] Eq. 4 · ALiBi bias

Each head gets its own slope m — different heads bias themselves to look at different effective distances. The slopes follow a geometric sequence (e.g. 2−8/H, 2−16/H, …, 2−8) so that with H heads you cover several orders of magnitude.

The killer property: ALiBi extrapolates well beyond the training context. Train on 1024 tokens, evaluate on 8192, and perplexity stays sensible. Sinusoidal PE and RoPE both struggle with this — they were never asked, during training, to behave well at unseen positions.

alibi.py 
def alibi_slopes(num_heads: int) -> torch.Tensor:
    # Geometric sequence: 2^(-8/H), 2^(-16/H), ..., 2^(-8)
    start = 2 ** (-8.0 / num_heads)
    return torch.tensor([start ** i for i in range(1, num_heads + 1)])


def alibi_bias(num_heads: int, seq_len: int) -> torch.Tensor:
    slopes = alibi_slopes(num_heads)                       # (H,)
    pos    = torch.arange(seq_len)                              # (N,)
    relpos = pos[None, :] - pos[:, None]                     # (N, N), j - i
    # For causal LMs we typically only allow j <= i, so |j - i| = i - j ≥ 0
    bias   = slopes[:, None, None] * relpos[None, :, :]      # (H, N, N)
    return bias

# Inside attention:
scores = Q @ K.transpose(-2, -1) / math.sqrt(d_k)
scores = scores + alibi_bias(H, N).to(scores.device)

RoPE

− Slight quality penalty vs. RoPE at fixed length.

Section 04 · Masks

Local and sparse attention — shaping the matrix.

The full N × N attention matrix is wasteful: in most natural sequences, a token's relevant context is local, or at least sparse. By masking entries we forbid certain query-key interactions — sometimes for correctness (causal masking), sometimes for efficiency (sliding window, block sparse).

How masking works

A mask is a boolean (or 0/1) matrix the same shape as the attention scores. Wherever the mask is zero, we set the score to -∞ before softmax. Softmax of -∞ is exactly zero, so those positions contribute nothing to the output.

masks.pyfour useful mask shapes 
def causal_mask(N: int) -> torch.Tensor:
    """Lower-triangular: position i can attend to positions 0..i."""
    return torch.tril(torch.ones(N, N, dtype=torch.bool))


def sliding_window_mask(N: int, window: int, causal: bool = True):
    """Position i attends to positions in [i-window, i] (causal)
    or [i-window, i+window] (bidirectional)."""
    i = torch.arange(N)[:, None]
    j = torch.arange(N)[None, :]
    if causal:
        return (j <= i) & (j >= i - window)
    return (j >= i - window) & (j <= i + window)


def dilated_mask(N: int, window: int, dilation: int = 2):
    """Sliding window with gaps (Longformer-style)."""
    i = torch.arange(N)[:, None]
    j = torch.arange(N)[None, :]
    rel = (i - j).abs()
    return (rel <= window * dilation) & (rel % dilation == 0)


def block_sparse_mask(N: int, block: int = 8, stride: int = 4):
    """Strided block pattern: local blocks plus periodic global blocks."""
    mask = torch.zeros(N, N, dtype=torch.bool)
    for i in range(N):
        # local block
        lo = max(0, (i // block) * block)
        hi = min(N, lo + block)
        mask[i, lo:hi] = True
        # global blocks every `stride`
        mask[i, ::stride] = True
    return mask

Causal masking and the autoregressive constraint

Language modelling is autoregressive: to predict token i+1, the model may only look at tokens 0..i. Causal masking enforces this at the attention level. Crucially, this is done during training with a single forward pass over the whole sequence — the lower-triangular mask makes every position predict the next one in parallel. At inference, we generate one token at a time, and the KV cache (Section 08) makes that efficient.

Sliding window — local attention

For long documents where most relevant context is nearby, sliding-window attention bounds each token's attention to ±w neighbours. Cost drops from O(N²) to O(N · w). Mistral's 7B uses sliding-window attention with w = 4096; combined with stacked layers, information still propagates across the full sequence (token i's output at layer 1 sees tokens i ± w, but at layer 2 it sees i ± 2w via those, and so on).

Block sparse patterns

These models are cousins, not copies. Longformer uses local windowed attention plus a task-motivated set of global tokens. BigBird adds random connections on top of local and global links. Sparse Transformer uses factorized fixed or strided sparse patterns. The shared idea is to keep only a carefully chosen subset of query-key interactions so the cost scales subquadratically with sequence length.

Don't confuse mask with bias

A mask zeroes the post-softmax weight (via -∞ pre-softmax). A bias (like ALiBi) shifts the score — every position still receives some weight. Masking is a hard binary; bias is a soft preference. They compose: ALiBi-on-causal is a causal mask plus a linear bias added to the surviving entries.

Section 05 · Sparsity

ProbSparse attention — most queries are lazy.

Sparse masks are designed in advance — you decide a pattern and stick with it. ProbSparse attention, introduced by the Informer paper, learns the sparsity from the data: for each query, decide at runtime whether it deserves full attention or can be replaced by a default.

The observation

Empirically, the attention distribution of most queries is close to uniform — that query is not picking anything out, so its softmax is roughly 1/N everywhere. Only a minority of queries have peaked, informative distributions. If we identify these "important" queries cheaply, we can run full attention only for them and approximate the rest with the average value vector.

The KL-divergence trick

The KL divergence between a query's softmax distribution and the uniform distribution measures how far that query is from being lazy. Computing it exactly costs as much as the original attention (O(NLK)). Informer's trick: estimate it from a random sample of U = c · ln(LK) keys. The estimator is the difference between the max and the mean of the sampled scores:

\[ \bar M(q_i) = \max_{j \in \mathrm{sample}} (q_i \cdot k_j) - \operatorname{mean}_{j \in \mathrm{sample}} (q_i \cdot k_j) \] Eq. 5 · sparsity score

A high M̄ means the query has a sharp peak somewhere in its distribution (the max stands out from the mean). Take the top-u = c · ln(LQ) queries by M̄, give them full attention, and replace the rest with the mean of V.

probsparse.pyInformer-style ProbSparse attention 
def prob_sparse_attention(Q, K, V, c=5):
    """
    Q, K, V: (B, H, N, d)  -- assume L_Q = L_K = N
    """
    B, H, N, d = Q.shape
    U = int(c * math.log(N))                              # number of sampled keys
    u = int(c * math.log(N))                              # number of top queries

    # 1. Sample U random key indices per query (in practice: same sample per head)
    sample_idx = torch.randint(N, (U,), device=Q.device)
    K_sample   = K[:, :, sample_idx, :]                       # (B, H, U, d)

    # 2. Sparsity score for every query: max - mean over sampled scores
    Q_K = Q @ K_sample.transpose(-2, -1) / math.sqrt(d)        # (B, H, N, U)
    M   = Q_K.max(dim=-1).values - Q_K.mean(dim=-1)         # (B, H, N)

    # 3. Pick top-u queries by sparsity
    top_idx = M.topk(u, sorted=False).indices                  # (B, H, u)
    Q_top   = torch.gather(Q, 2, top_idx[..., None].expand(-1, -1, -1, d))

    # 4. Full attention only for top queries — O(u · N · d)
    scores  = Q_top @ K.transpose(-2, -1) / math.sqrt(d)
    weights = F.softmax(scores, dim=-1)
    out_top = weights @ V                                       # (B, H, u, d)

    # 5. Default output for the rest: the mean V
    out = V.mean(dim=2, keepdim=True).expand(-1, -1, N, -1).clone()
    out.scatter_(2, top_idx[..., None].expand(-1, -1, -1, d), out_top)
    return out

Section 06 · Linearization

Linear attention — swap the parentheses.

Standard attention computes \( \operatorname{softmax}(QK^\top)V \). The \( N \times N \) matrix in the middle is the bottleneck. Linear attention attacks the problem by removing the softmax — and gains the freedom to re-associate the matrix product.

The kernel reformulation

Softmax is what couples queries and keys non-linearly — it's why we cannot just rearrange the matmul. Replace softmax with a feature map φ applied separately to Q and K:

\[ \operatorname{attn}(Q, K, V) = \frac{\phi(Q)\bigl(\phi(K)^\top V\bigr)} {\phi(Q)\sum_j \phi(K_j)} \] Eq. 6 · linear attention

The trick is the parenthesisation. Read it carefully: instead of computing \( \phi(Q)\phi(K)^\top \) first, which creates an \( N \times N \) matrix, we compute \( \phi(K)^\top V \) first, which creates only a \( d \times d \) matrix, and then multiply by \( \phi(Q) \) on the left. The matmul dimensions are exactly the same; the cost is wildly different.

What is φ?

The feature map must produce non-negative values, so the implicit kernel \( \phi(q)^\top \phi(k) \) remains a valid similarity. Common choices include \( \phi(x) = \operatorname{elu}(x) + 1 \) (Katharopoulos et al., the original "linear transformer"), random Fourier features (Performer), or polynomial maps. The quality of the approximation to softmax depends entirely on φ — a poor choice and the model just refuses to learn.

linear_attn.pyKatharopoulos et al. linear transformer 
def linear_attention(Q, K, V):
    """
    Q, K, V: (B, H, N, d).  Returns: (B, H, N, d).
    Complexity O(N d²) instead of O(N² d).
    """
    # Feature map: ELU + 1, ensures phi(x) > 0
    phi_Q = F.elu(Q) + 1                                # (B, H, N, d)
    phi_K = F.elu(K) + 1

    # Key trick: compute phi(K)^T @ V *first* — gives a (d, d) matrix
    KV = phi_K.transpose(-2, -1) @ V                  # (B, H, d, d)

    # Normaliser: sum over keys
    Z = phi_Q @ phi_K.sum(dim=-2, keepdim=True) \
                  .transpose(-2, -1)                    # (B, H, N, 1)
    return (phi_Q @ KV) / (Z + 1e-6)


# Causal version uses a running accumulator over time:
def causal_linear_attention(Q, K, V):
    phi_Q, phi_K = F.elu(Q) + 1, F.elu(K) + 1
    # einsum forms the (d, d) outer products at each time step,
    # then cumulative-sums them — sequential in N but O(d²) memory per step.
    KV = torch.einsum('bhnd,bhne->bhnde', phi_K, V).cumsum(dim=2)
    out = torch.einsum('bhnd,bhnde->bhne', phi_Q, KV)
    return out

Gated DeltaNet — the Qwen-style upgrade

Plain linear attention compresses history into a few running statistics, which is fast but can make precise overwrites hard: once old information has been blended into the accumulator, you do not get a clean way to say "replace that association with this new one". Gated DeltaNet changes the memory rule itself. Instead of only accumulating key-value summaries, it keeps a fixed-size associative state matrix S and performs an error-correcting update at each step.

Conceptually, the state predicts what value the current key should retrieve; the model measures the residual; then it writes only that residual back into memory, modulated by gates. This makes the state act less like a bag of all past tokens and more like a continually edited key-value memory. That is the big intuition behind why Gated DeltaNet tends to retrieve more sharply than older linear-attention variants.

\[ \begin{aligned} \hat{v}_t &= S_{t-1} k_t \\ e_t &= v_t - \hat{v}_t \\ S_t &= \alpha_t S_{t-1} + \beta_t\, e_t k_t^\top \\ o_t &= S_t q_t \end{aligned} \] Conceptual recurrence · gated delta attention / Gated DeltaNet

Read that recurrence slowly. The \(\alpha_t\) gate decides how much old memory survives. The \(\beta_t\) gate decides how hard we write the new correction. Decode-time memory stays fixed at the size of the state matrix instead of growing with a KV cache, so the per-token decode footprint is constant with respect to context length.

gated_deltanet_step.pypedagogical single-step update 
def gated_delta_step(q_t, k_t, v_t, S, alpha_t, beta_t):
    # q_t, k_t: (d_k,), v_t: (d_v,), S: (d_v, d_k)
    # In real models, q/k are usually normalized and gates come from learned projections.

    pred = S @ k_t                                  # what memory currently predicts for this key
    err  = v_t - pred                               # only write the residual

    S = alpha_t * S + beta_t * torch.outer(err, k_t)
    o_t = S @ q_t
    return o_t, S
Why Qwen uses it

Qwen's current hybrid line uses Gated DeltaNet as the linear-attention workhorse and periodically inserts ordinary softmax attention layers for exact global mixing. The resulting pattern is a practical compromise: most layers get constant-memory decode and long-context efficiency, while a minority of full-attention layers preserve the crisp retrieval and global interaction patterns that pure recurrent mixers still struggle with.

The catch

Linear attention's quality depends on the memory rule being expressive enough to preserve the right associations. In practice, vanilla linear attention often underperforms softmax attention on hard language tasks. Newer successors like Performer, Gated DeltaNet, and the recent Mamba / state-space family narrow that gap, but they do so by introducing more structure, more gates, and more implementation complexity. Linear-time sequence mixers remain most compelling when context is so long that a quadratic KV-cache story stops being practical.

Causal linear attention is subtle

Replacing softmax breaks the easy causal mask. In the causal version, you must accumulate KV sequentially — a running outer product over time. This sequential dependency removes one of linear attention's main advantages over RNNs: parallel training. Chunkwise tricks recover most of it, but the implementation is no longer pretty.

Section 07 · IO-aware

FlashAttention — do more arithmetic, win.

ProbSparse and linear attention are algorithmic approximations — they compute something different from full attention. FlashAttention is different: it computes exactly the same output as standard attention, but rearranges the computation to avoid ever writing the N × N matrix to memory.

The bottleneck nobody talked about

On modern GPUs, attention is bound not by compute but by memory bandwidth. The N × N score matrix is materialised in HBM (the slow global memory), then read back to apply softmax, then read again to multiply by V. For long sequences, the time spent moving the score matrix between HBM and the compute units dominates the actual arithmetic.

The trick: tile and stream

FlashAttention (Dao et al., 2022) processes one row-block of Q at a time. It loads that block into SRAM (fast on-chip memory), then streams through column-blocks of K and V. At each step it computes a partial dot product, updates a running softmax (more on this below), and updates a partial output. The full N × N matrix never exists anywhere except as transient tiles in SRAM.

Animation · tile streaming
FlashAttention's memory ballet
A Q-tile lives in SRAM throughout. K/V tiles stream past it. Running statistics (m, ℓ) and partial output O are kept in registers and updated incrementally.

Online softmax

The mathematical core of FlashAttention is the online softmax. Suppose we have already processed key-blocks 1..j-1 and have a running max m, a running normalisation ℓ, and a running (unnormalised) output O. We see a new block giving scores Sj. We update:

\[ \begin{aligned} m' &= \max\!\bigl(m,\max(S_j)\bigr), & \alpha &= e^{m-m'}, & P_j &= e^{S_j-m'} \\ \ell' &= \alpha\,\ell + \sum P_j, & O' &= \alpha\,O + P_j V_j \end{aligned} \] Eq. 7 · online softmax

The α factor rescales the previous partial output to match the new maximum, ensuring numerical stability and exact equivalence to single-pass softmax. After the last block, divide once by ℓ.

flash_sketch.pypedagogical PyTorch — NOT fast on GPU 
def flash_attention_naive(Q, K, V, block_size=64):
    """
    Mathematically identical to standard attention; demonstrates
    the tile structure. In real FlashAttention, this loop runs
    on the GPU with Q-tiles permanently in SRAM.
    """
    B, H, N, d = Q.shape
    out = torch.zeros_like(Q)

    # Outer loop: row-blocks of Q (independent — parallelisable)
    for i in range(0, N, block_size):
        Q_i = Q[:, :, i:i+block_size, :]                  # (B, H, Br, d)
        m = torch.full((B, H, Q_i.shape[2], 1), -float('inf'))
        l = torch.zeros((B, H, Q_i.shape[2], 1))
        O = torch.zeros_like(Q_i)

        # Inner loop: column-blocks of K/V (sequential — running state)
        for j in range(0, N, block_size):
            K_j = K[:, :, j:j+block_size, :]
            V_j = V[:, :, j:j+block_size, :]

            S_ij = Q_i @ K_j.transpose(-2, -1) / math.sqrt(d)

            m_new = torch.maximum(m, S_ij.max(dim=-1, keepdim=True).values)
            alpha = torch.exp(m - m_new)
            P     = torch.exp(S_ij - m_new)

            l = alpha * l + P.sum(dim=-1, keepdim=True)
            O = alpha * O + P @ V_j
            m = m_new

        out[:, :, i:i+block_size, :] = O / l
    return out

Why this is fast (in real implementations)

Real FlashAttention runs the outer loop in parallel across CUDA blocks (one block per Q-tile) and the inner loop sequentially within a block. The Q-tile stays in registers/SRAM for the entire kernel. K and V tiles are loaded into SRAM once per inner-loop iteration. Everything else lives in registers. The attention matrix is never written to HBM.

The result: 2×–4× speedup on standard attention, scaling to 10× on long sequences (where the HBM traffic was crushing). FlashAttention does more floating-point operations than naive attention (because of the rescaling) but does far fewer HBM round-trips. On a modern GPU, this is what matters.

The bigger lesson

FlashAttention won not because it found a better algorithm — the math is the same. It won because it asked the right question: "what is actually slow about this on a GPU?" and answered it. Most modern fused kernels (RMSNorm, RoPE, SwiGLU) share the same playbook.

Section 08 · KV efficiency

Grouped / Multi-Query / Latent Attention — share or compress the KV.

At inference time, autoregressive models cache K and V for all previous tokens — the "KV cache" — to avoid recomputing them at every step. For a long context with many heads, this cache dominates memory: a 32-layer MHA decoder with 32 heads, head-dim 128, and 8k context burns about 4 GB of KV cache per request in fp16/bf16. Grouped, Multi-Query, and Latent attention attack this bottleneck in two different ways: either share keys/values across query heads, or compress them into a learned latent bottleneck.

The implementation

GQA is structurally identical to MHA, with two small differences: Wk and Wv project to a smaller dimension (one entry per KV-group instead of one per head), and we repeat_interleave the K/V tensors before the attention multiply to broadcast each group across its query heads.

gqa.pygrouped-query attention 
class GroupedQueryAttention(nn.Module):
    def __init__(self, d_model, num_heads, num_kv_heads):
        super().__init__()
        assert num_heads % num_kv_heads == 0
        self.H, self.H_kv = num_heads, num_kv_heads
        self.d_k = d_model // num_heads
        self.group_size = num_heads // num_kv_heads

        # Note: W_k, W_v project to a *smaller* dim than W_q
        self.W_q = nn.Linear(d_model, num_heads    * self.d_k, bias=False)
        self.W_k = nn.Linear(d_model, num_kv_heads * self.d_k, bias=False)
        self.W_v = nn.Linear(d_model, num_kv_heads * self.d_k, bias=False)
        self.W_o = nn.Linear(d_model, d_model, bias=False)

    def forward(self, x, mask=None):
        B, N, _ = x.shape
        Q = self.W_q(x).view(B, N, self.H,    self.d_k).transpose(1, 2)
        K = self.W_k(x).view(B, N, self.H_kv, self.d_k).transpose(1, 2)
        V = self.W_v(x).view(B, N, self.H_kv, self.d_k).transpose(1, 2)

        # Repeat K/V to match Q's head count — each group serves `group_size` heads
        K = K.repeat_interleave(self.group_size, dim=1)
        V = V.repeat_interleave(self.group_size, dim=1)

        scores = Q @ K.transpose(-2, -1) / math.sqrt(self.d_k)
        if mask is not None:
            scores = scores.masked_fill(mask == 0, float('-inf'))
        attn = F.softmax(scores, dim=-1)

        out = (attn @ V).transpose(1, 2).contiguous().view(B, N, -1)
        return self.W_o(out)

How much do you save?

For Llama-2-70B (H = 64), switching to GQA with 8 KV-heads cuts the KV cache by 8× — from about 20 GB to about 2.5 GB for an 8k context, assuming fp16/bf16 storage. The model quality penalty is often modest enough that mainstream large models adopt it: Llama 2 70B uses GQA, and the Llama 3 family uses GQA across sizes. MQA pushes harder (1 KV-head) but with a larger quality drop, so it's mostly seen where inference efficiency matters more than preserving full MHA-style flexibility.

Multi-Head Latent Attention (MLA)

GQA and MQA save memory by making multiple query heads share the same K and V tensors. MLA, introduced by the DeepSeek team, goes one step further: instead of caching full per-head keys and values, it first compresses each token into a low-dimensional latent KV state, and reconstructs head-specific keys and values from that state when attention is computed.

This is the crucial distinction from MQA. In MQA, all query heads literally read the same key/value head. In MLA, heads can still end up with different reconstructed keys and values; the shared object is the latent bottleneck, not the final attention vectors. So MLA trades a bit more projection logic for a much smaller decode-time cache.

\[ c_t^{KV} = W^{DKV} h_t, \qquad k_t = W^{UK} c_t^{KV}, \qquad v_t = W^{UV} c_t^{KV} \] Eq. 9 · MLA compresses K/V through a latent state

Read this equation operationally. For token t, the model first forms a compact latent vector \(c_t^{KV}\). That is the thing it wants to keep around at inference time. When a future query needs to attend back to token t, the model expands that latent back into keys and values through learned up-projections. DeepSeek's official report presents MLA as the mechanism that "significantly compresses" the KV cache; in the DeepSeek-V2 release, they report a 93.3% KV-cache reduction relative to DeepSeek 67B.

mla.pypedagogical latent-KV attention sketch 
class MultiHeadLatentAttention(nn.Module):
    def __init__(self, d_model, num_heads, latent_dim):
        super().__init__()
        self.H = num_heads
        self.d_k = d_model // num_heads

        self.W_q   = nn.Linear(d_model, num_heads * self.d_k, bias=False)
        self.W_dkv = nn.Linear(d_model, latent_dim, bias=False)     # down-project into cacheable latent
        self.W_uk  = nn.Linear(latent_dim, num_heads * self.d_k, bias=False)
        self.W_uv  = nn.Linear(latent_dim, num_heads * self.d_k, bias=False)
        self.W_o   = nn.Linear(d_model, d_model, bias=False)

    def forward(self, x, mask=None):
        B, N, _ = x.shape
        Q = self.W_q(x).view(B, N, self.H, self.d_k).transpose(1, 2)

        c_kv = self.W_dkv(x)                                      # (B, N, r)  -- cache this at decode time
        K = self.W_uk(c_kv).view(B, N, self.H, self.d_k).transpose(1, 2)
        V = self.W_uv(c_kv).view(B, N, self.H, self.d_k).transpose(1, 2)

        scores = Q @ K.transpose(-2, -1) / math.sqrt(self.d_k)
        if mask is not None:
            scores = scores.masked_fill(mask == 0, float('-inf'))
        attn = F.softmax(scores, dim=-1)
        out = (attn @ V).transpose(1, 2).contiguous().view(B, N, -1)
        return self.W_o(out), c_kv
One practical wrinkle

Real MLA implementations are fussier than this pedagogical sketch. In practice, positional encoding interacts awkwardly with compression, so production MLA variants often keep a small dedicated positional path outside the main latent bottleneck, and aggressively fuse projections so reconstruction stays cheap. But the core idea does not change: cache the latent, not the full expanded per-head K/V tensors.

Why this matters for serving

An LLM serving system's throughput is mostly bounded by KV-cache memory: more memory per request = fewer concurrent requests. GQA cuts that cost by sharing K/V across heads; MLA cuts it harder by compressing them into a cacheable latent state. These are exactly the kinds of architectural changes that make long-context production LLMs economically viable.

Section 09 · Encoder-decoder

Cross attention — two sources, one query.

Self-attention has Q, K, V all coming from the same sequence. Cross attention separates them: Q from a "target" sequence (the decoder) and K, V from a "source" sequence (the encoder). It's how a decoder reads what an encoder produced.

Where you see it

The original Transformer's decoder has two attention layers per block: a self-attention over its own previous tokens, and a cross-attention into the encoder output. T5, BART, MarianMT — every encoder-decoder model uses this. Modern decoder-only LLMs (GPT, Llama) skip it entirely: there is no separate encoder. Cross attention remains essential for translation, summarisation, vision-language models (Q-Former, Flamingo's perceiver resampler), and any multi-modal architecture.

cross_attn.py 
class CrossAttention(nn.Module):
    def __init__(self, d_model, num_heads):
        super().__init__()
        self.H = num_heads
        self.d_k = d_model // num_heads
        self.W_q = nn.Linear(d_model, d_model, bias=False)
        self.W_k = nn.Linear(d_model, d_model, bias=False)
        self.W_v = nn.Linear(d_model, d_model, bias=False)
        self.W_o = nn.Linear(d_model, d_model, bias=False)

    def forward(self, x_target, x_source, mask=None):
        # Q from target (decoder state), K/V from source (encoder output)
        B, N_t, _ = x_target.shape
        B, N_s, _ = x_source.shape
        Q = self.W_q(x_target).view(B, N_t, self.H, self.d_k).transpose(1, 2)
        K = self.W_k(x_source).view(B, N_s, self.H, self.d_k).transpose(1, 2)
        V = self.W_v(x_source).view(B, N_s, self.H, self.d_k).transpose(1, 2)

        scores = Q @ K.transpose(-2, -1) / math.sqrt(self.d_k)
        if mask is not None:                              # typically a padding mask on source
            scores = scores.masked_fill(mask == 0, float('-inf'))
        attn = F.softmax(scores, dim=-1)                       # (B, H, N_t, N_s)

        out = (attn @ V).transpose(1, 2).contiguous().view(B, N_t, -1)
        return self.W_o(out)

Two sequence lengths, one matrix

Notice the shape of the attention matrix: (Nt, Ns). Target queries on the rows, source keys on the columns. The two sequences need not be the same length — translating "Eu te amo" (3 tokens) to "I love you" (3 tokens), or a 100-frame video to a 5-sentence caption. The decoder's i-th token attends to all source positions, picking up whichever ones it needs at that step of generation.

No causal mask is needed for cross attention itself — at decoding step i, all of the source is already known. (The decoder's self-attention is still causal.)

Section 10 · Self-attention types

Self-attention variants — who sees whom.

Self-attention is a family. The differences live entirely in the mask: where you allow attention, who is hidden, when. Three patterns cover almost everything in modern NLP.

Bidirectional (BERT-style)

No mask. Every token sees every other token, including the future. This is the right choice for representation learning when the full sequence is known up front — sentiment classification, span extraction, NLI. The training task that motivates it is masked language modelling (MLM): a few tokens are replaced with [MASK] and the model predicts them using both left and right context.

Causal (GPT-style)

Lower-triangular mask. Each position attends only to itself and earlier positions. This is what makes left-to-right next-token prediction sensible — at position i, the model has not "seen" position i+1, so its prediction is honest. Every modern decoder-only LLM (GPT, Llama, Mistral, Qwen) is causal.

Prefix LM

A hybrid. The first k tokens (the "prefix") attend bidirectionally; the remaining tokens attend causally to everything before them. Useful for tasks framed as conditional generation: the prefix is the input (a passage), the suffix is the output (a question's answer). T5 and UL2 use this. The mask looks like a causal lower-triangular with a fully-attended top-left block.

variants.pythe three masks side by side 
def bidirectional_mask(N):
    return torch.ones(N, N, dtype=torch.bool)


def causal_mask(N):
    return torch.tril(torch.ones(N, N, dtype=torch.bool))


def prefix_lm_mask(N, prefix_len):
    mask = torch.tril(torch.ones(N, N, dtype=torch.bool))      # causal everywhere
    mask[:prefix_len, :prefix_len] = True                       # fully attend on prefix
    return mask
A practical note

Causal masking has a small efficiency benefit beyond honesty: with FlashAttention, the lower-triangular structure lets the kernel skip half the score matrix entirely. The savings compound at long context lengths — one reason causal LMs are easier to train to 100k tokens than bidirectional ones.

Section 11 · Forecasting

Time-series attention — periods, scales, and channels.

Forecasting sequences are not just "text with numbers". The dependency that matters is often one day ago, one week ago, or this sensor at the same frequency as that one, not simply "whatever token is semantically similar". Time-series transformers therefore add three biases on top of vanilla attention: they look across multiple time scales, they often work in the frequency domain, and they explicitly model cross-channel coupling.

Vanilla full attention is still a sensible baseline, especially for short horizons or irregular event sequences. But for long forecasting windows it pays a quadratic price to compare every timestamp with every other timestamp, even when the real signal is dominated by seasonal lags, smooth trends, and a handful of recurring frequencies.

Multi-scale temporal

Run attention at several resolutions at once: one branch sees every timestep, another sees downsampled windows, another sees very coarse summaries. Fine scales catch spikes; coarse scales catch long seasonal structure.

Frequency-domain

Transform the series with an FFT and reason over frequency bins instead of raw timestamps. Long-range periodic structure becomes compact, and shifts in time become phase changes in the spectrum.

Cross-channel spectral

In multivariate forecasting, channels can lock together at some frequencies but decouple at others. Attention across channels per frequency bin exposes exactly that structure.

Notation shift

In time-series models, N is usually horizon length, C is the number of variables or sensors, and d is representation width. Channels are a property of the data; heads are a property of the architecture. They are not the same thing.

Multi-scale temporal attention

The most direct fix for long time windows is to stop asking one attention map to solve everything. At a fine scale you want short-range transients and local continuity; at a coarse scale you want daily, weekly, or monthly trends. Multi-scale temporal attention builds a pyramid of views over the same sequence and attends within each one.

Architecturally, this usually means downsample → attend → upsample → fuse. The coarse branches are cheaper because they work on fewer tokens, and they inject an inductive bias that says "long-range structure should be smooth enough to survive resampling".

multiscale_temporal.pycoarse-to-fine temporal attention 
class MultiScaleTemporalAttention(nn.Module):
    def __init__(self, d_model, scales=(1, 2, 4), window=32):
        super().__init__()
        self.scales = scales
        self.window = window
        self.blocks = nn.ModuleList([
            MultiHeadAttention(d_model, num_heads=4) for _ in scales
        ])
        self.mix = nn.Linear(d_model * len(scales), d_model)

    def forward(self, x):
        B, N, d = x.shape
        outs = []
        for s, attn in zip(self.scales, self.blocks):
            x_s = x if s == 1 else downsample(x, factor=s)     # (B, N/s, d)
            mask = sliding_window_mask(x_s.size(1), max(1, self.window // s))
            y_s, _ = attn(x_s, mask)
            y_s = y_s if s == 1 else upsample(y_s, size=N)   # back to original length
            outs.append(y_s)

        fused = torch.cat(outs, dim=-1)
        return self.mix(fused)

Frequency-domain attention

A seasonal signal that looks long-range in time often looks tiny in frequency space: a daily cycle, a weekly cycle, maybe a few harmonics. This is why many forecasting models first apply an FFT, then perform gating or attention over frequency bins rather than raw timesteps.

There are two distinct benefits. First, the spectrum can be much more compressible than the time series. Second, many transformations become easier to learn there: periodic amplification, damping, denoising, and lag reasoning all have cleaner geometric interpretations.

frequency_attention.pyattention over Fourier bins 
def spectral_attention(x):
    # x: (B, N, C) time domain; FFT along time
    Xf = torch.fft.rfft(x, dim=1)                          # (B, F, C), complex

    # Real-valued features for attention: concatenate real and imaginary parts
    feat = torch.cat([Xf.real, Xf.imag], dim=-1)         # (B, F, 2C)
    Q = W_q(feat); K = W_k(feat); V = W_v(feat)

    # Attend over frequency bins, not timestamps
    scores = Q @ K.transpose(-2, -1) / math.sqrt(Q.size(-1))
    weights = F.softmax(scores, dim=-1)              # (B, F, F)
    Y = weights @ V

    # Map back to a complex spectrum, then invert
    Yf = torch.complex(Y[..., :x.size(-1)], Y[..., x.size(-1):])
    return torch.fft.irfft(Yf, n=x.size(1), dim=1)
Important distinction

Going to the frequency domain does not automatically make a model subquadratic. If you still attend densely over all F frequency bins, the attention part is still quadratic in F. The win comes when the spectrum is either compressed, sparsified, or mixed with lighter operations than full attention.

Cross-channel spectral attention

Multivariate forecasting adds another axis: channels. A temperature sensor, a power meter, and a pressure sensor may be tightly coupled at a 24-hour rhythm, weakly coupled at a 1-hour rhythm, and independent in the high-frequency noise. Cross-channel spectral attention captures exactly that: it lets channels attend to one another within each frequency bin.

cross_channel_spectral.pychannel attention inside each frequency bin 
def cross_channel_spectral_attention(x):
    # x: (B, N, C).  Channels are the tokens; frequency bins are the contexts.
    Xf = torch.fft.rfft(x, dim=1)                              # (B, F, C)
    feat = torch.stack([Xf.real, Xf.imag], dim=-1)         # (B, F, C, 2)
    feat = proj(feat)                                      # (B, F, C, d_h)

    Q = W_q(feat); K = W_k(feat); V = W_v(feat)
    scores = torch.einsum('bfcd,bfed->bfce', Q, K) / math.sqrt(Q.size(-1))
    weights = F.softmax(scores, dim=-1)              # attention over channels, per frequency bin
    Y = torch.einsum('bfce,bfed->bfcd', weights, V)
    return Y                                               # inverse FFT omitted for focus

Autoformer-style autocorrelation attention

Autoformer makes a sharper forecasting claim: for many series, the useful dependency is not an arbitrary pairwise interaction between times i and j, but a small set of delays such as 24, 48, or 168. So instead of learning an N × N attention map, it measures which lags best align the query pattern with the key pattern, keeps only the top few lags, and aggregates shifted values.

\[ R_{QK}(\tau) = \operatorname{IFFT}\!\bigl(\operatorname{FFT}(Q)\cdot\overline{\operatorname{FFT}(K)}\bigr)_\tau \] Eq. 8 · autocorrelation from the convolution theorem

This is exact autocorrelation, computed in \( O(N \log N) \) with FFTs rather than \( O(N^2) \) pairwise comparisons. Once we have the lag scores, we keep only the strongest \( k \) delays and mix the correspondingly shifted value streams.

Interactive · seasonal lags
Autocorrelation discovers repeating delays
autoformer_autocorr.pyfind top-k seasonal delays 
def autocorrelation_lags(Q, K, top_k=4):
    # Q, K: (B, H, N, d)
    q = Q.mean(dim=-1)                                 # (B, H, N)
    k = K.mean(dim=-1)

    qf = torch.fft.rfft(q, dim=-1)
    kf = torch.fft.rfft(k, dim=-1)
    corr = torch.fft.irfft(qf * torch.conj(kf), n=q.size(-1), dim=-1)   # (B, H, N_lag)

    top_lags = corr.topk(top_k, dim=-1).indices
    lag_weights = F.softmax(torch.gather(corr, -1, top_lags), dim=-1)
    return top_lags, lag_weights

Vectorized autocorrelation attention

The paper idea is only half the story. A literal Python loop over delays throws away most of the GPU. The systems-friendly version builds all shifted copies in parallel with index arithmetic and torch.gather, so the top-k lag mixture becomes one batched tensor expression.

vectorized_autocorr.pyno Python loop over lags 
def vectorized_lag_mix(V, top_lags, lag_weights):
    # V: (B, H, N, d)
    # top_lags, lag_weights: (B, H, K)
    B, H, N, d = V.shape
    K = top_lags.size(-1)

    base = torch.arange(N, device=V.device)                         # (N,)
    idx = (base[None, None, None, :] - top_lags[..., None]) % N   # (B, H, K, N)

    V_exp = V[:, :, None, :, :].expand(-1, -1, K, -1, -1)      # (B, H, K, N, d)
    idx   = idx[..., None].expand(-1, -1, -1, -1, d)        # gather indices
    shifted = torch.gather(V_exp, 3, idx)                         # (B, H, K, N, d)

    weights = lag_weights[..., None, None]                       # (B, H, K, 1, 1)
    return (weights * shifted).sum(dim=2)                    # (B, H, N, d)

Vanilla full attention Arbitrary pairwise timestamp interactions O(N²d) Short windows, irregular events, weak periodicity Expensive and weakly biased toward seasonality
Multi-scale temporal attention Fine spikes + coarse trends can be modeled separately Σ_s O((N/s) · w_s · d) Mixed local and long-range structure Needs careful resampling and fusion design
Frequency-domain attention Seasonality is sparse in Fourier space O(N log N + F²d) or lighter Long smooth periodic components dominate Abrupt discontinuities are less natural spectrally
Cross-channel spectral attention Variables interact differently at different frequencies O(N log N + F C²) Multivariate sensors with phase-locked behavior Channel cost grows quadratically without grouping
Autoformer autocorrelation A few seasonal lags explain most dependencies O(N log N + kNd) Clear repeating periods, long horizons Less flexible when the regime changes suddenly
Vectorized autocorrelation Same lag bias as Autoformer, but GPU-friendly Same asymptotics, better constants You want the autocorrelation idea to actually run fast Implementation is more index-heavy than concept-heavy
Practical takeaway

For forecasting, the right question is often not "which timestep is most relevant?" but "which lag, scale, and frequency are most relevant?" The best time-series attention variants hard-code that question into the architecture.

Section 12 · Pattern gallery

Attention heatmaps — what do heads see?

Attention is famously interpretable — or at least, more interpretable than most parts of a neural network. The (N, N) weight matrix is a heatmap of "who looked at whom", and trained heads often display striking, human-readable patterns. These are synthetic but realistic patterns. Head 0 is "previous-token" — every query attends to its left neighbour. Head 1 is "syntactic" — verbs attend to their subjects. Head 2 is "diffuse" — broad context summary. Head 3 is "punctuation" — strong attention to commas/periods. Real trained heads look very much like this.

The reading-room test

If you only remember one thing about interpreting attention: patterns are heads talking, not the model talking. A single head can show a clear pattern, but the model's decision for token i is a sum over many heads across many layers, mixed by MLPs in between. A clean attention heatmap is suggestive of a hypothesis, not proof of one.

Section 13 · End-to-end

A mini transformer — everything together.

We close the tour by assembling everything into a tiny transformer you can train on a toy dataset in a notebook. Two layers, two heads, embedding dim 32. A sequence-copy task: the model sees a random sequence of digits and learns to repeat them.

mini_transformer.pyfull model in ~60 lines 
import torch, torch.nn as nn, torch.nn.functional as F, math


class TransformerBlock(nn.Module):
    def __init__(self, d_model, num_heads, ff_mult=4):
        super().__init__()
        self.attn  = MultiHeadAttention(d_model, num_heads)
        self.norm1 = nn.LayerNorm(d_model)
        self.norm2 = nn.LayerNorm(d_model)
        self.ff    = nn.Sequential(
            nn.Linear(d_model, ff_mult * d_model),
            nn.GELU(),
            nn.Linear(ff_mult * d_model, d_model),
        )

    def forward(self, x, mask):
        # Pre-LN residual structure (modern default)
        attn_out, attn_w = self.attn(self.norm1(x), mask)
        x = x + attn_out
        x = x + self.ff(self.norm2(x))
        return x, attn_w                              # return weights for visualisation


class MiniTransformer(nn.Module):
    def __init__(self, vocab_size, d_model=32, num_heads=2,
                 num_layers=2, max_len=32):
        super().__init__()
        self.tok_emb = nn.Embedding(vocab_size, d_model)
        self.pos_emb = nn.Embedding(max_len, d_model)
        self.blocks  = nn.ModuleList([
            TransformerBlock(d_model, num_heads) for _ in range(num_layers)
        ])
        self.norm    = nn.LayerNorm(d_model)
        self.head    = nn.Linear(d_model, vocab_size, bias=False)

    def forward(self, ids):
        B, N = ids.shape
        pos = torch.arange(N, device=ids.device)
        x = self.tok_emb(ids) + self.pos_emb(pos)[None]

        mask = torch.tril(torch.ones(N, N, dtype=torch.bool, device=ids.device))
        attn_maps = []
        for block in self.blocks:
            x, attn = block(x, mask)
            attn_maps.append(attn)

        return self.head(self.norm(x)), attn_maps


# -------- Training loop ------------------------------------------------
def train_copy_task():
    V, N = 10, 12                                       # vocab = digits 0-9, seq len 12
    model = MiniTransformer(V, max_len=N).cuda()
    opt   = torch.optim.AdamW(model.parameters(), lr=3e-3)

    for step in range(2000):
        x = torch.randint(0, V, (64, N), device='cuda')        # random sequences
        y = x.clone()                                       # target = same sequence
        logits, _ = model(x)
        loss = F.cross_entropy(logits.view(-1, V), y.view(-1))
        opt.zero_grad(); loss.backward(); opt.step()
        if step % 200 == 0: print(f"step {step:4d}  loss {loss.item():.4f}")
    return model


# -------- Extracting attention for visualisation ----------------------
model = train_copy_task()
test  = torch.tensor([[3, 1, 4, 1, 5, 9, 2, 6, 5, 3, 5, 8]], device='cuda')
_, attn_maps = model(test)
# attn_maps[layer] has shape (1, num_heads, N, N) — that's your heatmap.

What you should see

On the copy task, the loss drops from ~2.3 (random) to under 0.05 in a few hundred steps. Inspecting the learned attention maps: head 0 in layer 0 typically learns to attend to the same position (a soft identity), so the model can route each input token's embedding straight to the corresponding output position. Layer 1 then refines this with the previous-position and skip-position heads.

Modifications to try

Reverse the sequence — the model now needs to attend to N − i. Watch the attention pattern flip diagonally. Sort the sequence — sorting requires a much richer attention pattern, and a 2-layer model struggles. Bump to 4 layers and watch it learn anyway. Add positional encoding ablations — strip out pos_emb entirely and the model can no longer copy at all. The position information is doing real work.

From here, scale

This exact code, with d_model bumped to 768, num_layers to 12, num_heads to 12, and the input swapped to tokenised text, is a GPT-2 small. The training loop is the same. The architectural ideas are the same. Modern LLMs are bigger and use RoPE, GQA, FlashAttention, and SwiGLU — but the bones of the calculation are exactly what's above.
