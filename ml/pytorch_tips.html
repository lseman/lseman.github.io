I · Tensors
Dtypes, devices, and the things people get wrong first

Every PyTorch bug involving a type error or a device mismatch traces back to not understanding dtypes and the device abstraction from day one.

Tensor creation — the right idioms# Always specify dtype explicitly
x = torch.tensor([1.0, 2.0], dtype=torch.float32)

# zeros/ones/empty on the right device
device = "cuda" if torch.cuda.is_available() else "cpu"
w = torch.zeros(64, 128, device=device, dtype=torch.float32)

# ✗ DON'T: creates on CPU, then .to(device) copies it
w = torch.zeros(64, 128).to(device)   # extra alloc + copy

# ✓ DO: allocate directly where you need it
w = torch.zeros(64, 128, device=device)  # one allocation

# Tensor from numpy — shares memory (CPU only)
arr = np.ones((4, 4), dtype=np.float32)
t = torch.from_numpy(arr)   # zero-copy, shares buffer
arr[0, 0] = 99              # t[0,0] is also 99 !

# To break the link:
t = torch.from_numpy(arr).clone()
dtype hierarchy

float32 — default, safe everywhere. float16 / bfloat16 — AMP training. float64 — rarely needed, expensive. int64 — indices, class labels. bool — masks. Mixing dtypes silently promotes or errors depending on op.

II · Shapes
view, reshape, squeeze, unsqueeze, permute, expand — and the broadcasting rules

Shape bugs are the most common PyTorch mistake. Master these five operations and broadcasting and 80% of shape errors disappear.

The shape toolkitx = torch.randn(4, 8, 16)   # [B=4, T=8, D=16]

## view vs reshape
x.view(4, 128)        # fast, must be contiguous
x.reshape(4, 128)     # safe, copies if needed — prefer this
x.reshape(4, -1)      # -1 = infer from total numel

## squeeze / unsqueeze
x.unsqueeze(0)        # [1, 4, 8, 16] — add dim at 0
x.unsqueeze(-1)       # [4, 8, 16, 1] — add at end
x.squeeze(0)          # remove dim 0 IF size==1, else no-op
x.squeeze()           # removes ALL size-1 dims — dangerous!

## permute — reorder axes
x.permute(0, 2, 1)    # [4, 16, 8] — swap T and D
x.T                   # only for 2D, use permute for >2D

## expand vs repeat
bias = torch.zeros(1, 8, 16)
bias.expand(4, 8, 16) # zero-copy view (strides=0)
bias.repeat(4, 1, 1)  # physically copies data — avoid

## flatten / unflatten
x.flatten(1)          # [4, 128] — flatten from dim 1
x.unflatten(1,(2,4))  # [4, 2, 4, 16] — split dim 1
Broadcasting rules (memorize these)

PyTorch aligns shapes from the right. Two dims are compatible if they are equal or one of them is 1.

$[B, T, D] + [D] \;\longrightarrow\; [B, T, D]$ — bias add ✓

$[B, T, 1] \cdot [1, 1, D] \;\longrightarrow\; [B, T, D]$ — outer-like ✓

$[B, T] + [B, D]$ — error: T ≠ D and neither is 1 ✗

When in doubt: torch.broadcast_shapes(a.shape, b.shape)

Common pattern — attention mask broadcast 
mask = torch.ones(T, T).tril()     # [T, T]
# needs to broadcast over [B, H, T, T]
mask = mask.unsqueeze(0).unsqueeze(0)  # [1,1,T,T]
attn = attn.masked_fill(mask == 0, float('-inf'))
Debugging shapes: add print(x.shape) after every operation while developing. Switch to assert x.shape == (B, T, D), x.shape in production to catch regressions early.
view gotcha

view fails on non-contiguous tensors. Always use reshape unless you need the zero-copy guarantee.

permute gotcha

After permute, the tensor is non-contiguous. Call .contiguous() before view().

expand gotcha

expand gives stride-0 tensors. Writing to them is undefined behavior. Only use for reads.

III · einsum & Indexing
The two operations that unlock readable, fast tensor math

einsum replaces nested loops and confusing permute+matmul chains. Advanced indexing replaces Python for-loops over batch elements.

einsum — read it as a contraction spec# Matrix multiply: (B,M,K) × (B,K,N) → (B,M,N)
out = torch.einsum('bmk,bkn->bmn', A, B)
# same as: torch.bmm(A, B)

# Attention scores: (B,H,T,D) × (B,H,D,T) → (B,H,T,T)
scores = torch.einsum('bhid,bhjd->bhij', Q, K)

# Outer product: (N,) × (M,) → (N,M)
outer = torch.einsum('i,j->ij', a, b)

# Trace: (N,N) → scalar
tr = torch.einsum('ii->', A)

# Batch dot product: (B,D) × (B,D) → (B,)
dot = torch.einsum('bd,bd->b', x, y)

# Diagonal: (N,N) → (N,)
diag = torch.einsum('ii->i', A)
Advanced indexing — gather, scatter, index_select# gather: pick values at given indices
logits = torch.randn(B, T, V)         # [B,T,vocab]
idx    = torch.randint(0, V, (B, T))  # [B,T]
chosen = logits.gather(
    2, idx.unsqueeze(-1)              # [B,T,1]
).squeeze(-1)                         # → [B,T]

# boolean mask indexing
mask = x > 0.5                 # bool tensor, any shape
vals = x[mask]                 # 1-D, variable length — no grad path!

# fancy indexing (grad-safe)
rows = torch.tensor([0, 2])
x[rows, :]                     # [2, D]
How to read an einsum string

Each letter is an axis. Axes on the left of → are contracted (summed over) if they don't appear on the right. Axes on the right of → survive in the output.

'bik,bkj->bij': $b$ and $i,j$ survive; $k$ is summed — this is batched matrix multiply.

'bii->b': $i$ is both a row and column index (diagonal), summed — this is batched trace.

Performance note

Under the hood, torch.einsum calls cuBLAS/BLAS for most contractions. For large tensors it is as fast as an explicit matmul. Use opt_einsum (or torch.einsum with optimize=True in PyTorch ≥ 2.0) when contracting three or more tensors — it finds the optimal contraction order, which can be orders-of-magnitude faster.

IV · Autograd
How gradients actually work — and five ways to break them silently

PyTorch autograd builds a computation graph dynamically. Understanding it lets you debug silent gradient bugs, control memory, and write custom differentiable ops.

The computation graphx = torch.randn(4, requires_grad=True)
y = x ** 2 + 2 * x             # graph: y → x
loss = y.mean()
loss.backward()                 # fills x.grad
print(x.grad)                   # dy/dx = 2x + 2

# MUST zero grads between steps — they accumulate!
optimizer.zero_grad()           # or set_to_none=True (faster)

## Detaching — stop gradient flow
target = model_ema(x).detach()  # EMA target, no grad
z = x.detach().numpy()          # safe numpy extraction

## no_grad context — inference, evaluation
with torch.no_grad():
    pred = model(x)             # no graph built, faster

## inference_mode — even faster (PyTorch ≥1.9)
with torch.inference_mode():
    pred = model(x)

## retain_graph — only when you need to call
## backward() more than once on same graph
loss.backward(retain_graph=True)
5 silent gradient killers:
1 Converting to NumPy mid-forward (.numpy() detaches silently)
2 In-place ops on leaf tensors with requires_grad=True — raises error or corrupts graph
3 Not zeroing grads → they accumulate across backward calls
4 float(x) or x.item() inside forward — detaches the value
5 Copying a tensor with = instead of .clone() — shared storage, in-place writes corrupt history
The gradient tape mental model

Each operation records itself into a function node in the graph. backward() walks the graph in reverse, calling each node's gradient function (chain rule). The graph is freed after backward() unless retain_graph=True.

Memory cost: the graph holds intermediate activations needed for the backward pass. With large models, this is the dominant memory consumer during training.

x.grad_fn

The function that created x. None means it's a leaf. Inspect to debug graph structure.

x.requires_grad

Only leaf tensors need this set. It propagates to all downstream ops automatically.

torch.autograd.grad

Compute gradients without accumulating into .grad — useful for Hessians and meta-learning.

V · Training Loop
The canonical loop — every line has a reason

There is one correct order for the six operations in a training step. Getting any one wrong produces bugs that are hard to notice until evaluation.

The canonical PyTorch training stepmodel.train()          # sets Dropout/BN to train mode

for batch in dataloader:
    x, y = batch
    x, y = x.to(device), y.to(device)

    # ── STEP 1: zero gradients ──────────────────
    optimizer.zero_grad(set_to_none=True)
    # set_to_none=True: frees memory instead of
    # writing zeros → slightly faster

    # ── STEP 2: forward pass ────────────────────
    logits = model(x)              # graph builds here

    # ── STEP 3: compute loss ────────────────────
    loss = criterion(logits, y)

    # ── STEP 4: backward pass ───────────────────
    loss.backward()                # fills .grad on params

    # ── STEP 5: (optional) gradient clipping ────
    torch.nn.utils.clip_grad_norm_(
        model.parameters(), max_norm=1.0
    )

    # ── STEP 6: optimizer step ──────────────────
    optimizer.step()               # updates parameters

    # ── STEP 7: scheduler step ──────────────────
    scheduler.step()   # call AFTER optimizer.step()
!

zero_grad BEFORE forward

Grads accumulate. If you forget, gradients from the previous batch add to this one — model converges to the wrong solution silently.

!

model.train() vs model.eval()

Dropout is active only in train mode. BatchNorm uses running stats in eval mode. Forgetting model.eval() during validation gives wrong loss numbers.

!

scheduler.step() order

In PyTorch ≥ 1.1, call scheduler.step() after optimizer.step(). Doing it before causes a one-step offset in the LR schedule.

!

clip_grad_norm_

Returns the total gradient norm before clipping — log it. Consistently high norms signal instability. Set max_norm=1.0 as a safe default for most transformers.

Always wrap validation in with torch.no_grad() and call model.eval(). Both matter. no_grad skips the graph; eval() changes layer behavior.

VI · Optimizers
Optimizer choices, weight decay gotcha, and LR scheduling

AdamW is not just Adam with weight decay — the difference matters. And your LR schedule is as important as your optimizer choice.

AdamW — the correct way# Adam adds weight decay to the gradient (wrong)
# Adam: θ ← θ - lr*(m̂/√v̂) - lr*λ*(m̂/√v̂)  ← couples!

# AdamW decouples weight decay from adaptive update
# AdamW: θ ← θ*(1-lr*λ) - lr*(m̂/√v̂)  ← correct!

optimizer = torch.optim.AdamW(
    model.parameters(),
    lr=3e-4,
    betas=(0.9, 0.95),  # β2=0.95 for transformers
    weight_decay=0.1,   # 0.01–0.1 for transformers
    fused=True,         # fused CUDA kernel, ~20% faster
)

## Don't apply weight decay to biases/LayerNorm!
no_decay = ['bias', 'norm']
params = [
    {'params': [p for n,p in model.named_parameters()
               if not any(nd in n for nd in no_decay)],
     'weight_decay': 0.1},
    {'params': [p for n,p in model.named_parameters()
               if any(nd in n for nd in no_decay)],
     'weight_decay': 0.0},
]
optimizer = torch.optim.AdamW(params, lr=3e-4)
LR schedulers — the most useful ones## Cosine annealing with linear warmup (transformer standard)
from torch.optim.lr_scheduler import (
    LinearLR, CosineAnnealingLR, SequentialLR
)
warmup = LinearLR(optimizer, start_factor=0.01,
                  end_factor=1.0, total_iters=500)
cosine = CosineAnnealingLR(optimizer, T_max=10000, eta_min=1e-5)
scheduler = SequentialLR(optimizer,
    schedulers=[warmup, cosine], milestones=[500])

## OneCycleLR — great for fast experiments
scheduler = torch.optim.lr_scheduler.OneCycleLR(
    optimizer, max_lr=3e-4,
    steps_per_epoch=len(loader), epochs=10
)

## ReduceLROnPlateau — validation-aware
scheduler = torch.optim.lr_scheduler.ReduceLROnPlateau(
    optimizer, mode='min', patience=3, factor=0.5
)
scheduler.step(val_loss)  # pass metric, not step count
Gradient clipping + AdamW: clip before optimizer.step(). Log torch.nn.utils.clip_grad_norm_'s return value — if the norm is always exactly 1.0 you're clipping every step, which is a sign your LR is too high.
SGD+momentum

Still best for vision CNNs with careful LR tuning. AdamW wins on transformers.

Muon / Shampoo

Second-order optimizers gaining traction in 2024–25. Worth watching for LLM training.

param groups

Use separate LR for backbone vs head: [{'params':..., 'lr':1e-5}, {'params':..., 'lr':1e-3}]

VII · AMP
Automatic mixed precision — 2× speed, same accuracy

Mixed precision runs the forward pass in float16/bfloat16 and keeps a float32 master copy for the optimizer. Three lines of code, significant speedup.

AMP with GradScaler — the full patternfrom torch.amp import autocast, GradScaler

scaler = GradScaler("cuda")

for x, y in loader:
    optimizer.zero_grad(set_to_none=True)

    ## autocast: ops run in float16 where safe
    with autocast("cuda", dtype=torch.float16):
        logits = model(x)
        loss   = criterion(logits, y)

    ## GradScaler: multiply loss to avoid fp16 underflow
    scaler.scale(loss).backward()

    ## unscale before clipping so norms are in fp32 space
    scaler.unscale_(optimizer)
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)

    ## step + update scale factor
    scaler.step(optimizer)
    scaler.update()
    scheduler.step()

## bfloat16: no scaler needed (wider exponent range)
with autocast("cuda", dtype=torch.bfloat16):
    loss = criterion(model(x), y)
loss.backward()   # no scaler!
Why float16 needs GradScaler but bfloat16 doesn't

float16: 5-bit exponent, range $\approx [6\times10^{-5},\, 65504]$. Small gradients underflow to zero. GradScaler multiplies loss by a scale factor $S$ before backward, then divides by $S$ after — keeping gradients in representable range.

bfloat16: 8-bit exponent (same as float32), range matches float32. No underflow problem. Preferred on A100/H100. Lower precision (7-bit mantissa vs 10) but fine for most training.

FormatExponent bitsMantissa bitsNeeds scaler?Best hardware
float32823NoAll GPUs
float16510YesV100, RTX
bfloat1687NoA100, H100, TPU
Key rule: LayerNorm and loss functions should stay in float32. autocast handles this automatically — it has a whitelist of ops that run in fp16 and a blacklist that stays fp32 (softmax, layer norm, loss).
On A100/H100: use dtype=torch.bfloat16 and drop the scaler entirely. On older GPUs (V100, RTX 20/30): use float16 + scaler. Check with torch.cuda.get_device_name().

VIII · Grad Accum
Simulating large batch sizes without the VRAM

Gradient accumulation lets you train with an effective batch of 512 on a GPU that can only fit 32. The trick is combining it correctly with AMP and normalization layers.

Gradient accumulation — correct patternaccum_steps = 16           # effective batch = micro × accum
micro_batch  = 32
# effective batch size = 32 × 16 = 512

optimizer.zero_grad(set_to_none=True)

for step, (x, y) in enumerate(loader):
    x, y = x.to(device), y.to(device)

    ## disable sync on intermediate steps (DDP)
    is_last = (step + 1) % accum_steps == 0
    ctx = model.no_sync() if not is_last else nullcontext()

    with ctx:
        with autocast("cuda", dtype=torch.bfloat16):
            loss = criterion(model(x), y)

        ## normalize loss by accum_steps so gradient
        ## magnitude matches a single large-batch step
        loss = loss / accum_steps
        scaler.scale(loss).backward()
        # grads ACCUMULATE (not zeroed until is_last)

    if is_last:
        scaler.unscale_(optimizer)
        torch.nn.utils.clip_grad_norm_(
            model.parameters(), 1.0
        )
        scaler.step(optimizer)
        scaler.update()
        scheduler.step()
        optimizer.zero_grad(set_to_none=True)
Why divide loss by accum_steps?

Gradients from backward() accumulate additively. After $K$ micro-batches the accumulated gradient equals $\sum_{k=1}^K \nabla_k$. To match a single batch of size $K \cdot m$, each term should contribute $\frac{1}{K}$ of its nominal value. So scale the loss (equivalently the gradient) by $\frac{1}{K}$.

$$\tilde{\nabla} = \frac{1}{K}\sum_{k=1}^K \nabla_{L_k} \approx \nabla_{L_{\text{full batch}}}$$

BatchNorm + gradient accumulation = broken. BN computes statistics per micro-batch, not the effective large batch. Use LayerNorm or GroupNorm instead when accumulating. This is one reason transformers prefer LayerNorm.
DDP note: model.no_sync()

In distributed training, each backward() triggers an all-reduce. With gradient accumulation, you want to all-reduce only on the last micro-step. model.no_sync() suppresses the sync on intermediate steps — without it you pay $K$ all-reduces instead of 1.

Memory management — avoiding OOM## Check VRAM usage
print(torch.cuda.memory_summary(device))
allocated = torch.cuda.memory_allocated()   # bytes in use

out = checkpoint(block, x)  # recomputes activations in backward
# use_reentrant=False is safer in modern PyTorch
out = checkpoint(block, x, use_reentrant=False)
Memory breakdown during training

For a model with $P$ parameters and batch size $B$, sequence length $T$:

Parameters (fp32): $4P$ bytes
Gradients (fp32): $4P$ bytes
Optimizer state (AdamW, fp32): $8P$ bytes (m + v)
Activations: $\sim O(B \cdot T \cdot D \cdot L)$ bytes
Total: $\approx 16P + \text{activations}$

For a 7B model: just parameters+optimizer ≈ $16 \times 7 \times 10^9 / 10^9$ = 112 GB. That's why you need 8-bit Adam, LoRA, or ZeRO.

Gradient checkpointing cuts activation memory by $\sim\sqrt{L}$ at the cost of one extra forward pass per layer during backward. Enable with model.gradient_checkpointing_enable() in HuggingFace, or manually with torch.utils.checkpoint.checkpoint.
OOM triage

Print torch.cuda.memory_summary() at the crash point. Check for tensor accumulation in lists inside the training loop.

tensor leak

Storing loss in a list for logging without .item() keeps the entire compute graph alive — classic OOM cause.

PYTORCH_CUDA_ALLOC_CONF

Set max_split_size_mb=512 to reduce fragmentation when OOM despite apparent free memory.

X · Debug: NaNs
Finding where NaNs enter your model — a systematic protocol

NaNs propagate forward and backward, so the first NaN you see is never where the bug is. Here is the systematic protocol to find the source in minutes.

NaN detection and localization## 1. Enable anomaly detection (slow but precise)
torch.autograd.set_detect_anomaly(True)
## Will print the exact forward op that produced NaN
## Disable in production — 2–3× slowdown

## 2. Register forward hooks on every layer
def nan_hook(module, inp, out):
    if isinstance(out, torch.Tensor):
        if torch.isnan(out).any():
            raise RuntimeError(
                f"NaN in {module.__class__.__name__}")

for layer in model.modules():
    layer.register_forward_hook(nan_hook)

## 3. Check inputs before forward pass
assert not torch.isnan(x).any(), "NaN in input batch"
assert not torch.isinf(x).any(), "Inf in input batch"

## 4. Check loss before backward
if torch.isnan(loss):
    print("NaN loss at step", step)
    print("logits stats:", logits.min(), logits.max())
    raise ValueError

## 5. Check gradients after backward
for name, p in model.named_parameters():
    if p.grad is not None and torch.isnan(p.grad).any():
        print(f"NaN grad in {name}")
Most common NaN sources:
log(0) — cross-entropy on zero-probability logit. Fix: add epsilon or use log_softmax
0/0 — division by near-zero norm. Fix: norm = norm.clamp(min=1e-8)
sqrt(0) — in layer norm or attention. Fix: add epsilon inside sqrt
exp(x) overflow — large logits in softmax. Fix: F.softmax handles this; manual implementations don't
fp16 overflow — value >65504 in fp16. Fix: use bfloat16 or add gradient clipping
Safe implementations of common numerically unstable ops

F.cross_entropy(logits, y) — use this, not manual log(softmax). It does the log-sum-exp trick internally.

F.normalize(x, dim=-1, eps=1e-8) — safe L2 norm with epsilon guard.

torch.linalg.norm(x, dim=-1, keepdim=True).clamp(min=1e-8) — explicit safe norm.

For attention: mask with -1e4 not -inf in fp16 — -inf can produce NaN in softmax when all positions are masked.

Protocol: (1) anomaly detection → find which layer; (2) inspect that layer's inputs/outputs; (3) identify the op (log, div, sqrt, exp); (4) add numerical guard. The anomaly detection message includes a stack trace — read it.

XI · Debug: Gradients
Reading gradient health — the signals that tell you training is wrong

Exploding gradients, vanishing gradients, and dead neurons each have a distinct signature in the gradient histogram. Know the patterns before they waste a week of training.

Gradient monitoring utilities## Per-layer gradient norm logging (log to wandb/tb)
def log_grad_norms(model, writer, step):
    for name, p in model.named_parameters():
        if p.grad is not None:
            writer.add_histogram(
                f"grad/{name}", p.grad, step)
            writer.add_scalar(
                f"grad_norm/{name}",
                p.grad.norm().item(), step)

## Quick inline gradient check
for name, p in model.named_parameters():
    if p.grad is None:
        print(f"No grad: {name}")   # disconnected!
        continue
    g = p.grad
    print(f"{name:40s} norm={g.norm():.3e}"
          f" max={g.abs().max():.3e}"
          f" frac_zero={( g==0).float().mean():.2f}")

## backward hook on a specific tensor
def save_grad(name):
    def hook(grad):
        grads[name] = grad.detach()
    return hook

grads = {}
x.register_hook(save_grad('x'))
loss.backward()
print(grads['x'].norm())

Healthy gradient norms for transformers are typically in $[10^{-3}, 10^{-1}]$ per layer. The first few layers (near the input) and last few (near the output) often have larger norms — that's normal. A sudden spike in a specific layer is the signal to investigate.
TensorBoard tip: log gradient histograms every 100 steps. A healthy histogram looks roughly Gaussian and stable over time. A histogram that collapses to a spike at zero means dead weights.

What torch.compile actually does

1. TorchDynamo traces your Python code, capturing the computation graph while handling Python control flow via "graph breaks."

2. TorchInductor compiles the graph to optimized CUDA/Triton kernels, fusing ops (e.g. linear + bias + activation → one kernel), eliminating redundant memory loads.

3. First call is slow (compilation). Subsequent calls use the cached compiled graph.

When torch.compile breaks: Python data-dependent control flow (e.g. if x.sum() > 0) causes graph breaks. Dynamic shapes (variable sequence lengths) cause recompilation. Use torch.compile(dynamic=True) for variable-length inputs.

XIII · Custom Modules
Writing clean, re-usable nn.Module subclasses

nn.Module is the fundamental building block. Know what it tracks automatically, what you have to register explicitly, and the common pitfalls.

nn.Module anatomyclass MyBlock(nn.Module):
    def __init__(self, d_model, n_heads):
        super().__init__()   # REQUIRED — never skip

        ## nn.Parameter: tracked by .parameters()
        self.scale = nn.Parameter(torch.ones(d_model))

        ## nn.Module: tracked by .modules() / .children()
        self.attn = nn.MultiheadAttention(d_model, n_heads)
        self.norm = nn.LayerNorm(d_model)
        self.ff   = nn.Linear(d_model, 4 * d_model)

        ## nn.ModuleList: list of modules — DO use this
        self.layers = nn.ModuleList([
            nn.Linear(d_model, d_model) for _ in range(4)
        ])

        ## WRONG: plain list is NOT tracked!
        self.bad = [nn.Linear(d, d) for _ in range(4)]

        ## register_buffer: non-param state (e.g. masks)
        mask = torch.ones(64, 64).tril()
        self.register_buffer('causal_mask', mask)
        ## self.causal_mask moves with .to(device) automatically

    def forward(self, x):
        x = self.norm(x)
        return x * self.scale   # scale is on correct device
Registration methodIncluded inMoves with .to()
nn.Parameterparameters(), state_dict()Yes
nn.Module attrmodules(), state_dict()Yes
nn.ModuleList/Dictmodules(), state_dict()Yes
register_bufferstate_dict(), not parameters()Yes
plain Python listNothingNo — moves break!
plain tensor attrNothingNo — stays on CPU!
Always use register_buffer for tensors that are not parameters but need to live on the same device as the model — positional encodings, causal masks, running statistics. Plain self.mask = mask will stay on CPU when you call model.to(device).
Use nn.ModuleDict when you have a variable number of named sub-modules (e.g. expert layers in MoE). Keys are accessible as self.experts['expert_0'] and are properly tracked.

XVI · Init & Norms
Weight initialization and normalization — the silent training-stability foundations

Bad initialization causes vanishing/exploding gradients before training even begins. Wrong normalization makes deep networks nearly untrainable.

Weight initialization — the right choicesimport torch.nn.init as init

## PyTorch defaults (often good enough)
## nn.Linear: Kaiming Uniform (fan_in mode)
## nn.Embedding: Normal(0, 1)
## nn.LayerNorm: weight=1, bias=0

## Xavier/Glorot — good for tanh/sigmoid activations
init.xavier_uniform_(layer.weight)
init.xavier_normal_(layer.weight)
## var = 2/(fan_in + fan_out)

## Kaiming/He — designed for ReLU/GELU
init.kaiming_uniform_(layer.weight, mode='fan_in',
                       nonlinearity='relu')
## var = 2/fan_in

## GPT-2 style: smaller init for residual projections
## std = 0.02 / sqrt(2 * n_layers)
std = 0.02 / (2 * n_layers) ** 0.5
init.normal_(proj.weight, mean=0, std=std)
init.zeros_(proj.bias)

## Zero-init output projection → identity at init
init.zeros_(block.mlp.c_proj.weight)
Xavier vs Kaiming — the key formula

Xavier: $\text{Var}[w] = \dfrac{2}{n_\text{in} + n_\text{out}}$. Designed to preserve variance through linear + symmetric activation (tanh).

Kaiming: $\text{Var}[w] = \dfrac{2}{n_\text{in}}$. Designed for ReLU which kills half of activations — doubles the variance to compensate.

For GELU/SiLU (not exactly ReLU), Kaiming is still a good default. The key property: variance of activations should stay $\approx 1$ through all layers at initialization.
