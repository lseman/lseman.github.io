Why sparse experts exist at all

Dense FFNs make capacity and active compute grow together.

In a standard transformer block, attention mixes tokens and the FFN transforms every token with the same dense MLP. As models scale, that FFN often becomes a major source of parameters and FLOPs.

  • If we double FFN width, every token pays the full compute cost, even if only a small subset of patterns need that extra capacity.
  • Rare structures such as code syntax, tables, formulas, or uncommon domain words may benefit from specialized pathways rather than one universal MLP.
  • MoE separates total parameters from active per-token computation.
dense FFN conditional compute specialization
The core bargain is simple: keep many expert FFNs in memory, but run only a few of them for each token.
Capacity scaling sketch
More experts raise total capacity. If top-k stays fixed, active compute stays comparatively flat.
Dense

Every token runs one shared FFN.

MoE

Every token runs only its selected experts.

Tradeoff

Routing quality and communication now matter.

III · Math

Equations and tensor bookkeeping

The math is short. The real complexity is shapes, capacity, and renormalization.

Router logits

$x_t \in \mathbb{R}^{d}$

$r_t = W_r x_t \in \mathbb{R}^{N}$

$S_t = \operatorname{TopK}(r_t, k)$

Selected gates

$\hat g_{t,i} = \operatorname{softmax}(r_{t,S_t})_i$

$y_t = \sum_{i \in S_t} \hat g_{t,i} E_i(x_t)$

Capacity per expert

$C = \left\lceil c_f \cdot \frac{T \cdot k}{N} \right\rceil$

with sequence tokens $T$, experts $N$, top-k routing $k$, and capacity factor $c_f$.

Symbol Shape Meaning
$x$ $[B,T,d]$ input token representations
$r$ $[B,T,N]$ router logits over experts
$S$ $[B,T,k]$ top-k expert indices
$\hat g$ $[B,T,k]$ renormalized selected gates
$E_i(x)$ $[\le C,d]$ batched tokens processed by expert $i$
Top-1

Cheaper communication, simpler dispatch, sharper specialization.

Top-2

Smoother gradients and redundancy, but more all-to-all traffic.

Noise

Jitter or noisy logits can reduce early router collapse.

The gating softmax is usually applied only over the selected experts, not all experts after masking.

Forward pass and training logic

Routing is conceptually simple even though distributed kernels are not. 

Forward pass pseudocodefor each token x_t:
    logits[t] = router(x_t)
    idx[t] = topk(logits[t], k)
    gates[t] = softmax(logits[t][idx[t]])

for each expert e:
    bucket[e] = {tokens t where e in idx[t]}
    x_e = gather(x_t for t in bucket[e])
    y_e = expert_e(x_e)

for each token t:
    y_t = 0
    for slot, e in enumerate(idx[t]):
        y_t += gates[t, slot] * scatter(y_e, t, e)

return y

aux_loss = n_experts * sum(importance * load)
loss = task_loss + alpha * aux_loss + z_loss
Logical order

Score, pick, bucket, run expert MLPs, then scatter back.

Kernel reality

Production systems fuse gather, permutation, expert GEMMs, and scatter.

Why balancing

If one expert gets all traffic, MoE degenerates into a dense bottleneck.

Educational pseudocode follows token logic. High-performance MoE code is mostly about minimizing token movement and maximizing expert batch efficiency.

Minimal educational implementation

A small PyTorch layer is enough to demonstrate top-k routing. 

PyTorch: minimal MoE layerimport torch
import torch.nn as nn

class TopKMoE(nn.Module):
    def __init__(self, d_model, d_hidden, n_experts, k=2):
        super().__init__()
        self.k = k
        self.n_experts = n_experts
        self.router = nn.Linear(d_model, n_experts, bias=False)
        self.experts = nn.ModuleList([
            nn.Sequential(
                nn.Linear(d_model, d_hidden),
                nn.GELU(),
                nn.Linear(d_hidden, d_model),
            )
            for _ in range(n_experts)
        ])

    def forward(self, x):
        b, t, d = x.shape
        flat = x.reshape(b * t, d)
        logits = self.router(flat)
        topv, topi = torch.topk(logits, self.k, dim=-1)
        gates = topv.softmax(dim=-1)

        out = flat.new_zeros(flat.shape)
        for e, expert in enumerate(self.experts):
            pos, slot = (topi == e).nonzero(as_tuple=True)
            if pos.numel() == 0:
                continue
            y = expert(flat[pos])
            out.index_add_(0, pos, y * gates[pos, slot, None])

        return out.view(b, t, d), logits, topi
Shapes

Input: [B, T, d]
Router logits: [B*T, N]
Top-k indices: [B*T, k]

Why this version is useful

It is short, readable, and shows the exact ideas: router scores, top-k selection, per-expert token buckets, and gated recombination.

Important limitation

The Python loop over experts is pedagogical. Real MoE systems use grouped dispatch, fused kernels, and often all-to-all communication across devices. 

Example usagex = torch.randn(8, 128, 512)
moe = TopKMoE(512, 2048, n_experts=8, k=2)
y, logits, topi = moe(x)

Balancing and overflow control

Training needs an auxiliary loss and a capacity rule, not just a router. 

PyTorch: load balancing lossimport torch.nn.functional as F

def load_balance_loss(logits, topi, n_experts):
    probs = logits.softmax(dim=-1)
    importance = probs.mean(dim=0)

    dispatched = F.one_hot(topi, n_experts).float()
    load = dispatched.sum(dim=1).mean(dim=0) / topi.size(1)

    return n_experts * torch.sum(importance * load)

task = criterion(y, target)
aux = 1e-2 * load_balance_loss(logits, topi, moe.n_experts)
loss = task + aux

IX · Block Design

Where MoE sits inside the transformer

MoE usually replaces the dense FFN sublayer, while attention stays dense and global.

Attention mixes information across tokens. MoE changes the token-wise transformation after that mixing step.
Attention

Decides which tokens exchange information.

Dense FFN

Would apply the same MLP to every token.

MoE FFN

Applies different expert pathways to different tokens.

Key insight

MoE and attention are complementary. Attention handles interaction structure; experts handle computation specialization. This is why an MoE model can still use ordinary transformer attention.

A useful mental model is: attention is the communication layer, while MoE is the conditional processing layer.

The systems view

Dispatch groups tokens by expert, shuffles them across devices, runs GEMMs, then scatters back.

Routing is a permutation problem: gather → all-to-all → expert GEMMs → scatter back.

offsets  = expert_prefix_counts(topi)

Batch size

Larger batches fill expert GEMMs — small batches leave GPUs idle.

top-2 cost

Each token duplicates into two streams, doubling dispatch traffic.

Interconnect

All-to-all dominates step time when NVLink / IB bandwidth is limited.

Kernel efficiency

Grouped GEMMs must be well-tuned; naive loops kill throughput.

A model can be theoretically sparse yet practically slow. MoE wins only when routing overhead is small relative to expert compute.

Throughput

Failure signature

Entropy crashes, load histogram spikes, overflow stays high.

Healthy signature

Entropy declines gradually while expert utilization stays broad.

Common fix

Increase aux weight, add router noise, or raise capacity factor.

XII · Variants
13 / 14
Different routing choices, different tradeoffs

Design variants trade stability, redundancy, communication, and simplicity.

Variant Main idea Strength Cost
Top-1 routing Each token uses one expert only. Cheapest dispatch and least duplication. Can be brittle if routing mistakes are hard to recover from.
Top-2 routing Each token uses two experts and mixes both outputs. More redundancy and smoother routing gradients. More communication and more active compute.
Shared expert hybrid One dense shared expert runs for all tokens plus sparse experts. Guarantees a common pathway for every token. Partly gives back dense compute cost.
Capacity-limited Each expert has a token budget per batch. Predictable memory and communication. Dropped or rerouted overflow tokens can hurt quality.
Choose top-1 when

You need maximal efficiency and have robust balancing control.

Choose top-2 when

You want more stable specialization and can afford extra routing cost.

Choose hybrid when

You want a safety net so every token always gets a dense shared computation path.

The “best” MoE variant is often a systems decision as much as a modeling one. The router, network fabric, and batch shape all influence which design wins.

XII-A · Auxiliary Loss
Making load balancing precise

The auxiliary loss has two terms — importance and load — and a z-loss that stabilises router logits.

Importance term $$f_i = \frac{1}{T}\sum_{t=1}^{T} \mathbf{1}[i \in \text{top-}k(r_t)], \qquad P_i = \frac{1}{T}\sum_{t=1}^{T} \text{softmax}(r_t)_i$$ $$\mathcal{L}_\text{balance} = \alpha \cdot N \sum_{i=1}^{N} f_i \cdot P_i$$
Z-loss (router logit regulariser) $$\mathcal{L}_z = \beta \cdot \frac{1}{T}\sum_{t=1}^{T}\left(\log \sum_{i=1}^{N} e^{r_{t,i}}\right)^2$$
$\mathcal{L}_z$ keeps logits from growing large, which stabilises softmax and prevents routers from committing too hard too early. ST-MoE (2022) showed it substantially reduces training instabilities.
Reading the terms
  • $f_i$ — fraction of tokens routed to expert $i$ (hard, discrete).
  • $P_i$ — mean soft probability assigned to expert $i$ (differentiable).
  • $f_i \cdot P_i$ — large when expert $i$ both wins tokens AND gets high confidence. Penalising the product keeps both terms modest.
  • $N \cdot \sum f_i P_i$ — normalised so a perfectly balanced router gives 1.0 regardless of $N$.
α too small

Router collapses: two or three experts absorb all traffic.

α too large

Routing becomes near-uniform; expert specialisation is suppressed.

Typical α

0.01 in Switch Transformer; tune with the diagnostic curves from slide XI.

XII-B · Router Init
Getting training started right

Router initialisation and the first few thousand steps determine whether expert specialisation ever emerges.

1
Small weight initialisation. Initialise the router weight $W_r$ with a small standard deviation (e.g. 0.01). Large initial logits cause one expert to dominate immediately — specialisation that is hard to undo because gradient signal for losing experts vanishes.
2
Temperature annealing. Divide logits by temperature $\tau > 1$ early in training to soften routing and encourage exploration. Anneal $\tau \to 1$ over the first 5–10% of steps.
3
Jitter noise. Add uniform noise $\epsilon \sim \mathcal{U}(0, 1/N)$ to logits before top-$k$. Breaks symmetry cheaply and prevents deterministic early collapse without requiring any hyperparameter schedule.
4
Separate learning rate for the router. Routers often benefit from a 10× smaller learning rate than expert FFN weights. The router should adapt slowly relative to the experts it directs. 
Router forward with jitter + temperaturedef route(self, x, training=False):
    logits = x @ self.W_r          # [B, T, N]
    if training:
        noise = torch.rand_like(logits)
        noise = noise / self.n_experts
        logits = logits + noise    # jitter
    logits = logits / self.tau     # temperature
    gates = logits.softmax(-1)
    topi = logits.topk(self.k, -1).indices
    return gates, topi

Early collapse is permanent

If the router commits all tokens to expert 0 in the first 500 steps, the other experts receive zero gradient for their FFN weights. Restarting from scratch is often the only fix — not adjusting α.

Watch the per-expert token count histogram in the first 1000 steps. If one bar is already 10× the others, intervene early.

XII-C · Overflow
What happens when an expert is full

Capacity is a hard ceiling — tokens that exceed it must be dropped, rerouted, or padded.

Expert capacity $$C = \left\lceil c_f \cdot \frac{T \cdot k}{N} \right\rceil$$
$T$ = tokens in batch · $k$ = experts per token · $N$ = total experts · $c_f$ = capacity factor (typically 1.0–2.0)
StrategyWhat happens to overflow tokensTrade-off
Drop Token skips the expert; output is a pass-through of the residual. Fast, zero extra memory. Gradient silent on dropped tokens.
Reroute Token is sent to its next-best expert (top-$k$+1). No information loss, but creates load cascades.
Pad & mask Expert slot is padded; mask prevents its contribution. Keeps tensor shapes fixed; wastes compute on padding.
Expert choice Each expert picks its own top-$C$ tokens; no overflow by design. Token may be picked by 0 or multiple experts — needs post-hoc normalisation.
Capacity factor intuition

With perfect balance, each expert sees exactly $T k / N$ tokens. $c_f = 1.0$ allows no slack — any imbalance causes drops. $c_f = 1.25$ gives 25% headroom at the cost of 25% more memory allocation per expert buffer.

Drop-on-overflow is the standard in Switch Transformer and most production systems. Rerouting is appealing but creates second-order load spikes that can be worse than the original imbalance.

XII-D · Scaling
When does sparsity pay off?

MoE wins when parameter count matters more than per-token FLOPs — and only when routing overhead is small.

The core trade

A dense model with $P$ parameters uses $P$ parameters and $P$-worth of FLOPs per token. An MoE with the same active parameters but $N\times$ more total parameters can use a much larger capacity without extra per-token compute — at the cost of routing, memory, and communication.

Effective parameter count vs. active FLOPs $$\text{Active FLOPs} \approx \underbrace{2 \cdot d \cdot d_\text{ff} \cdot k}_{\text{expert FFNs}} + \underbrace{4 d^2}_{\text{attention}}$$
Active FLOPs scale with $k$, not $N$. Total parameters scale with $N$.
Empirically, MoE models match dense models at 3–5× lower training compute when $N \geq 8$ and the routing overhead is under 10% of step time. Below 8 experts the advantage shrinks considerably.
MoE wins when…

Parameters are the bottleneck for quality and memory per GPU is large enough to shard experts.

MoE breaks even when…

Routing overhead ≥ 15% of step time, or expert count is so small that load balancing is near-trivial.

Dense wins when…

Inference latency matters more than throughput, or the serving batch is too small to fill expert buffers.

Inference is harder than training

At training time, large batch sizes keep experts busy. At inference with batch size 1, each expert step is a tiny GEMM — often slower than the equivalent dense matmul due to routing and memory access patterns.

Rule of thumb

MoE is most attractive with $N \in \{8, 16, 32, 64\}$ experts, $k \in \{1, 2\}$, capacity factor 1.0–1.25, and a serving batch size large enough that every expert processes at least a few tokens per forward pass.

XIII · Practice
14 / 14
What matters in real systems

In production, the hardest parts are balancing traffic and moving tokens efficiently.

Good fits
  • Very large language models where FFNs dominate parameters.
  • Workloads with diverse token families and enough data for specialization.
  • Clusters where expert parallelism and all-to-all are efficient enough.
Failure modes
  • Router collapse: the same experts receive most tokens.
  • Overflow: experts exceed capacity and drop useful traffic.
  • Communication bottlenecks dominate saved compute.
Important knobs
  • top-k choice, usually 1 or 2
  • capacity factor and overflow policy
  • auxiliary loss weight, router noise, expert count
Practical reading of the layer

Attention answers which tokens interact. MoE answers which computation each token receives afterward. They solve different problems inside the same transformer block.

Bottom line

MoE is not a free lunch. It trades dense arithmetic for routing decisions, token shuffling, and systems complexity. When that trade is engineered well, model capacity can increase much faster than active per-token compute.

Useful mental model: MoE is a sparse replacement for the transformer MLP, not a replacement for attention.
