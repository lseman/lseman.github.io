I · Motivation
Why sparse experts exist at all

Dense FFNs couple capacity and active compute. MoE breaks that link.

In a standard transformer block, every token runs the same dense MLP. Doubling FFN width means every token pays the full compute cost — even if only a fraction of tokens need that capacity.

  • Rare token families (code syntax, formulas, rare vocabulary) may benefit from specialized pathways not shared with common prose.
  • MoE separates total parameter count from active per-token FLOPs by routing each token to a small subset of experts.
  • The core bargain: keep many expert FFNs in memory, but compute only $k \ll N$ of them per token.
conditional compute sparse activation expert specialization parameter efficiency
Key equation: $\text{Active FLOPs} \propto k$ while $\text{Total Parameters} \propto N$. Routing quality determines whether this bargain is honoured.
Capacity scaling sketch
More experts raise total capacity. Active compute stays flat when top-k is fixed.
Dense

Every token runs one shared FFN — capacity ≡ active compute.

MoE

Each token runs k out of N experts — capacity decoupled from compute.

Cost

Routing decisions, communication, and load balancing now matter greatly.

II · Formal Setup
Notation and problem statement

One MoE layer replaces the dense FFN sublayer. The math fits in four lines.

MoE layer — general form

Let $x_t \in \mathbb{R}^d$ be token $t$, with $N$ expert networks $\{E_i\}_{i=1}^N$.

$$y_t = \sum_{i=1}^{N} G(x_t)_i \cdot E_i(x_t)$$

$G(x_t) \in \mathbb{R}^N$ is the gate vector. Sparsity is achieved by making most $G(x_t)_i = 0$.

Top-k sparse gate

$$G(x_t) = \operatorname{TopK}\!\bigl(\operatorname{softmax}(W_r x_t),\; k\bigr)$$

$W_r \in \mathbb{R}^{N \times d}$ is the router weight. $\operatorname{TopK}$ keeps the $k$ largest values and zeros the rest, then renormalises over the selected set.

Renormalised gates (Switch / GShard style)

$$\hat{G}_{t,i} = \frac{\exp(r_{t,i})}{\sum_{j \in S_t} \exp(r_{t,j})}, \quad S_t = \operatorname{topk\text{-}indices}(r_t, k)$$

Symbol Shape Meaning
$x$ $[B,T,d]$ Input token representations
$W_r$ $[N,d]$ Router (gate) weight matrix
$r_t$ $[N]$ Router logits for token $t$
$S_t$ $[k]$ Indices of selected experts
$\hat{G}_t$ $[k]$ Renormalised gate weights
$E_i(x)$ $[d]$ Expert $i$ output for token $x$
$y_t$ $[d]$ MoE layer output for token $t$
$N$ scalar Total number of experts
$k$ scalar Active experts per token
When $k=N$ and all gates are non-zero, MoE recovers the dense FFN as a special case.

III · Token → Expert
The full forward pass, step by step

Token-to-expert routing: score, select, dispatch, compute, merge.

Solid lines = active paths. Dashed = inactive. Only 2 of 4 experts execute.
1

Score

Compute router logits $r_t = W_r x_t \in \mathbb{R}^N$. One linear projection, no nonlinearity.

2

Select

$\operatorname{TopK}(r_t, k)$ returns indices $S_t$ and values. Softmax is recomputed only over $S_t$ for the final gate weights $\hat{G}$.

3

Dispatch

Tokens are grouped by assigned expert into batches. Tokens routed to the same expert are packed together and sent (all-to-all in multi-device settings).

4

Compute

Each expert $E_i$ runs its FFN on its token batch: $h_i = W_2^{(i)}\,\sigma(W_1^{(i)} x)$. Experts execute in parallel.

5

Merge

Expert outputs scatter back to token order. $y_t = \sum_{i \in S_t} \hat{G}_{t,i} \cdot E_i(x_t)$. Weighted sum over selected experts only.

V · Linear Router Math
The default router — deep dive

The linear router is a single matmul. Its failure modes are subtle.

Step 1 — logit computation

$$r_t = W_r x_t, \quad W_r \in \mathbb{R}^{N \times d}$$

No bias, no nonlinearity. The router sees only the current token representation after the preceding layer norm.

Step 2 — top-k selection and renormalization

$$S_t = \operatorname{argtopk}(r_t, k)$$

$$\hat{G}_{t,i} = \frac{e^{r_{t,i}}}{\sum_{j \in S_t} e^{r_{t,j}}}, \quad i \in S_t, \quad \hat{G}_{t,i}=0 \text{ otherwise}$$

Noisy variant (Shazeer 2017)

$$r_t^{\text{noisy}} = W_r x_t + \epsilon \cdot \operatorname{softplus}(W_n x_t), \quad \epsilon \sim \mathcal{N}(0,1)$$

$W_n \in \mathbb{R}^{N \times d}$ learns per-expert noise scale. Disabled at inference.

Why renormalize over selected experts only?

If softmax is applied to all $N$ logits and then the losers are zeroed, the gate weights no longer sum to 1 — the output magnitude shrinks with $N$. Renormalizing over $S_t$ preserves the magnitude invariant: $\sum_{i \in S_t} \hat{G}_{t,i} = 1$.

Gradient flow through discrete top-k

The $\operatorname{argtopk}$ operation is not differentiable. Gradients flow only through the gate values (the softmax), not through the routing decision itself. This means the router learns which logits to make large, not which expert to pick directly.

Collapse mechanism

If expert $i$ gets a slightly higher initial logit, it receives more tokens → larger gradient signal → even higher logit. This positive feedback loop concentrates routing on 1–2 experts if not countered by an auxiliary loss or noise.

VI · Expert-Choice Router
Inverted routing — experts choose tokens

Expert-choice flips the selection: each expert picks its own top-C tokens.

Expert-choice formulation (Zhou et al. 2022)

Compute a score matrix $S \in \mathbb{R}^{N \times T}$:

$$S_{i,t} = \operatorname{softmax}_t(x_t^\top W_r^{(i)})$$

Each expert $i$ selects its top-$C$ tokens:

$$\mathcal{T}_i = \operatorname{argtopk}_t(S_{i,\cdot},\; C)$$

Output for token $t$ aggregated over all experts that selected it:

$$y_t = \sum_{i:\, t \in \mathcal{T}_i} S_{i,t} \cdot E_i(x_t)$$

Capacity is exact — no overflow by definition

$C = \lfloor c_f \cdot T / N \rfloor$ tokens per expert. Every expert processes exactly $C$ tokens. No auxiliary loss needed for load balance.

Token-choice vs Expert-choice
Left: tokens compete for experts — load imbalance possible. Right: experts compete for tokens — load guaranteed equal.
Trade-off: perfect load balance but some tokens may be selected by 0 experts (they receive a pass-through of the residual) or by many experts. Normalisation becomes non-trivial.

VII · Soft MoE
Continuous routing — no discrete decisions

Soft MoE replaces token dispatch with differentiable slot mixing. Zero dropped tokens.

Soft MoE — Puigcerver et al. 2023

Each expert $i$ has $p$ slots. For slot $s$ of expert $i$, form a weighted combination of all input tokens:

$$\tilde{x}_{i,s} = \sum_{t=1}^{T} \Phi(x,\Theta)_{t,(i,s)} \cdot x_t$$

where $\Phi(x,\Theta)_{t,(i,s)} = \operatorname{softmax}\!\left(\frac{x_t^\top \theta_{i,s}}{\sqrt{d}}\right)$ over all tokens $t$.

Run each expert on its slots: $\tilde{y}_{i,s} = E_i(\tilde{x}_{i,s})$.

Combine back to token space:

$$y_t = \sum_{i,s} \Phi(x,\Theta)_{t,(i,s)} \cdot \tilde{y}_{i,s}$$

$\theta_{i,s} \in \mathbb{R}^d$ are learned slot embeddings (router parameters). The dispatch and combine matrices are the same softmax, ensuring a clean gradient path.
Property Top-k Sparse Soft MoE
Routing Discrete, argmax Continuous, softmax
Dropped tokens Yes (overflow) Never
Expert load Uneven (needs aux loss) Determined by slot count
Gradient through router Partial (only gate values) Full end-to-end
Active compute $k/N$ fraction All experts (not sparse!)
Specialization Hard token assignment Soft mixture of tokens

Soft MoE is not computationally sparse

Every expert processes $p$ slots, and every slot is a mixture of all tokens. The benefit is training stability and the ability to use it as a plug-in replacement in ViTs and language models without the engineering complexity of token dispatch.

Google Brain (2023) showed Soft MoE outperforms sparse MoE at the same parameter count on ImageNet classification, with no routing instabilities.

VIII · Hash Router
Parameter-free routing

Hash routing assigns tokens to experts by a fixed function — no collapse, no gradient.

Hash router (Roller et al. 2021)

Assign token $t$ with vocabulary index $v_t$ to expert:

$$i^* = h(v_t) \bmod N$$

where $h$ is a hash function (e.g. Murmur3, random permutation). No learned parameters in the router at all.

Balanced hash variant

Pre-compute a random permutation $\pi$ of $\{0,\ldots,V-1\}$:

$$i^*(v_t) = \left\lfloor \frac{\pi(v_t) \cdot N}{V} \right\rfloor$$

Ensures each expert handles exactly $V/N$ vocabulary entries. Balance across tokens depends on vocabulary frequency distribution.

Hash routing is the theoretical lower bound on routing quality — pure positional assignment with no content awareness. Its value is as a baseline and for ablation.
Advantages
  • Zero router parameters — smaller model size.
  • No collapse by construction — no auxiliary loss needed.
  • Deterministic at inference — reproducible routing.
  • Zero routing overhead at forward time.

Fundamental limitation

Expert assignment is independent of token context. A word like "bank" always goes to the same expert regardless of whether it means a financial institution or a riverbank. Content-adaptive routing consistently outperforms hash routing on downstream benchmarks once the model is large enough to learn meaningful specialization.

Practical use case

Hash routing is valuable early in training as a warm-start to prevent collapse, then replaced by a learned router after the experts have developed initial specialization.

IX · Expert → Token
The return path — scatter and aggregate

Expert-to-token: how outputs scatter back to their original positions and weights.

Full dispatch-and-combine tensor view

Let $X \in \mathbb{R}^{T \times d}$ be the flattened token matrix ($T = B \cdot L$).

Dispatch matrix $D \in \{0,1\}^{T \times (N \cdot C)}$: $D_{t,(i,s)}=1$ iff token $t$ is in slot $s$ of expert $i$.

$$\tilde{X}_i = D_i^\top X \in \mathbb{R}^{C \times d} \quad \text{(gather)}$$

$$\tilde{Y}_i = E_i(\tilde{X}_i) \in \mathbb{R}^{C \times d} \quad \text{(expert FFN)}$$

Combine matrix $C \in \mathbb{R}^{T \times (N \cdot C)}$ holds the gate weights:

$$Y = \sum_{i=1}^{N} C_i \tilde{Y}_i \in \mathbb{R}^{T \times d} \quad \text{(scatter + weight)}$$

$D$ and $C$ share the same sparsity pattern. In practice they are fused into a single permutation + scale operation on the token buffer.
Gather → Expert → Scatter
Gather permutes tokens into expert order. Scatter reverses the permutation, adding gate-weighted outputs back.
top-1

Each token contributes to exactly one output bucket. Scatter is injective.

top-2

Each token is duplicated into two buckets. Scatter sums two contributions per token.

expert-choice

A token may appear in 0, 1, or multiple buckets. Normalisation required.

X · Capacity & Aux Loss
Training objectives that enforce balance

Three losses keep routing stable: load balance, z-loss, and capacity gating.

Expert capacity & load-balance loss (Switch / GShard)

$C = \lceil c_f \cdot Tk/N \rceil$ — tokens over $C$ are dropped. $c_f \in [1.0, 2.0]$.

$$f_i = \frac{1}{T}\sum_t \mathbf{1}[i \in S_t], \quad P_i = \frac{1}{T}\sum_t \operatorname{softmax}(r_t)_i$$

$$\mathcal{L}_\text{balance} = \alpha \cdot N \sum_{i=1}^N f_i \cdot P_i$$

$f_i$ is the hard token fraction (no gradient). $P_i$ is the differentiable proxy. Their product is large only when expert $i$ both wins tokens and gets high gate confidence.

Z-loss (ST-MoE 2022)

$$\mathcal{L}_z = \beta \cdot \frac{1}{T}\sum_t \!\left(\log \sum_i e^{r_{t,i}}\right)^{\!2}$$

Penalises large logit magnitudes. Prevents softmax saturation and training instability spikes. $\beta \approx 10^{-3}$.

Full training objective

$$\mathcal{L} = \mathcal{L}_\text{task} + \alpha\,\mathcal{L}_\text{balance} + \beta\,\mathcal{L}_z$$

Typical: $\alpha = 10^{-2}$, $\beta = 10^{-3}$. $N \cdot \sum f_i P_i = 1$ at perfect balance regardless of $N$.

α too small

Collapse: 1–2 experts absorb all tokens.

α too large

Uniform routing; no specialization emerges.

z-loss absent

Logit explosion → instability spikes mid-training.

Capacity factor guide
  • $c_f = 1.0$ — no slack, any imbalance drops tokens.
  • $c_f = 1.25$ — 25% headroom, recommended starting point.
  • $c_f = 2.0$ — near-zero drops, 2× expert buffer memory.
  • Monitor overflow rate; reduce $c_f$ once routing stabilises.
Loss-free alternative (DeepSeek-V3): per-expert bias $b_i$ adjusted by sign-gradient each step. Eliminates $\alpha$ entirely — see slide XVII.

XI · Switch Transformer
Fedus, Zoph & Le — Google 2021

Switch simplifies to top-1 routing and proves sparse MoE can match dense models at lower compute.

Switch routing (top-1 special case)

$$i^* = \arg\max_i\, \operatorname{softmax}(W_r x_t)_i$$

$$y_t = \operatorname{softmax}(W_r x_t)_{i^*} \cdot E_{i^*}(x_t)$$

The gate weight is the softmax probability of the chosen expert — still over all $N$ experts (not renormalised over a subset, because $k=1$).

Key design choices
  • Top-1 routing — simpler dispatch, half the all-to-all traffic vs. top-2.
  • Selective precision — router operates in float32; experts can be bfloat16.
  • Expert parallelism — each expert lives on a different device; dispatch is a single all-to-all call.
  • Capacity factor $c_f = 1.25$ during training, 2.0 during fine-tuning.
Switch-C: 1.6 trillion parameters, 2048 experts. Achieves 7× speedup over a T5-XXL at the same compute budget, setting a precedent for trillion-parameter sparse models.
Switch Transformer block
Every-other-layer MoE is common in practice. Switch used MoE in all FFN layers.

Mistral AI 2024 — sparse mixture of Mistral 7B experts

Mixtral uses top-2 routing over 8 experts per layer. Only 2 of 8 experts execute per token.

Mixtral gating (top-2)

$$y_t = \sum_{i \in \operatorname{top2}(W_g x_t)} \operatorname{softmax}(W_g x_t)_i \cdot \operatorname{SwiGLU}_i(x_t)$$

$W_g \in \mathbb{R}^{8 \times 4096}$. Each expert is a SwiGLU FFN with hidden dim 14336. Softmax is over all 8 logits, then the top-2 are renormalised.

XIII · DeepSeekMoE
DeepSeek AI 2024 — fine-grained expert decomposition

DeepSeekMoE: many small experts + shared experts, each expert ¼ the standard size.

DeepSeekMoE output

$$y_t = \sum_{i=1}^{K_s} E_i^{(s)}(x_t) + \sum_{i \in \mathcal{T}_t} G_{t,i} \cdot E_i^{(r)}(x_t)$$

$K_s$ shared experts always execute (dense, small). $\mathcal{T}_t \subset \{1,\ldots,N_r\}$ are the top-$K_r$ routed experts from a pool of $N_r$ fine-grained experts.

Fine-grained decomposition

Split each standard expert into $m$ fine-grained experts, each with $1/m$ of the FFN hidden dimension:

$$d_{\text{ff}}^{\text{fine}} = \frac{d_{\text{ff}}^{\text{standard}}}{m}$$

Increase active count to $k' = k \cdot m$ to keep active FLOPs equal. Total experts $N' = N \cdot m$. This gives more routing choices without increasing active compute.

DeepSeek-V2: $N_r = 160$ routed experts, $K_r = 6$ active, $K_s = 2$ shared. Each expert is ¼ the size of a standard FFN expert. Total ~236B params, ~21B active.
Why shared + routed?

Shared experts act as a "common knowledge" base that every token benefits from. Routed experts provide specialization on top of this base. This prevents the failure mode where a routed expert must cover both general and specialized content.

DeepSeekMoE architecture
Shared experts (always active) + top-6 of 160 routed fine-grained experts.

XV · PyTorch
Minimal educational implementation

Top-k MoE layer in ~30 lines. Router noise and aux loss included. 

TopKMoE — forward + aux lossimport torch, torch.nn as nn, torch.nn.functional as F

class TopKMoE(nn.Module):
    def __init__(self, d, d_ff, n, k=2, alpha=1e-2):
        super().__init__()
        self.k, self.n, self.alpha = k, n, alpha
        # router: small init avoids early collapse
        self.router = nn.Linear(d, n, bias=False)
        nn.init.normal_(self.router.weight, std=0.01)
        self.experts = nn.ModuleList([
            nn.Sequential(
                nn.Linear(d, d_ff), nn.GELU(),
                nn.Linear(d_ff, d))
            for _ in range(n)])

    def forward(self, x, training=False):
        B, T, d = x.shape
        flat = x.reshape(B*T, d)
        logits = self.router(flat)          # [B*T, N]
        if training:                        # jitter noise
            noise = torch.rand_like(logits) / self.n
            logits = logits + noise
        topv, topi = logits.topk(self.k, dim=-1)
        gates = topv.softmax(-1)            # renorm over top-k
        out = flat.new_zeros(flat.shape)
        for e, exp in enumerate(self.experts):
            pos, slot = (topi == e).nonzero(as_tuple=True)
            if pos.numel() == 0: continue
            out.index_add_(0, pos,
                exp(flat[pos]) * gates[pos, slot, None])
        aux = self._balance(logits, topi)
        return out.view(B, T, d), aux

    def _balance(self, logits, topi):
        # f_i * P_i load balance loss
        P = logits.softmax(-1).mean(0)
        f = F.one_hot(topi, self.n).float()\
              .sum(1).mean(0) / self.k
        return self.alpha * self.n * (f * P).sum()

What each part does
  • std=0.01 init — prevents early one-expert domination.
  • Jitter noise — breaks symmetry, encourages exploration.
  • Renorm over top-k — gates sum to 1 regardless of N.
  • index_add_ — sparse scatter; tokens accumulate into output buffer.
  • _balance — auxiliary loss penalises load concentration.

Pedagogical only

Production MoE uses grouped GEMMs, fused kernels, and all-to-all communication. The Python loop over experts is O(N) serial calls — functional but not performant.

All-to-all

Two all-to-all ops per MoE layer: one to send tokens to experts, one to return outputs.

Batch size

Larger batches fill expert GEMMs. Small batch → idle GPU cycles dominate.

Interconnect

NVLink / InfiniBand bandwidth caps MoE speedup. All-to-all saturates it first.

Kernel fusing

Production: gather + GEMM + scatter are fused. Naive loops kill throughput.

A model can be theoretically sparse yet practically slow. MoE wins only when routing overhead < 10–15% of step time.

XVII · Loss-Free Balance
DeepSeek-V3 innovation — 2025

Auxiliary-loss-free load balancing via per-expert bias terms in the router.

Biased top-k selection (DeepSeek-V3)

Add a learnable bias $b_i$ to each expert logit only for routing decisions:

$$S_t = \operatorname{argtopk}_i(r_{t,i} + b_i,\; k)$$

The gate weights use the unbiased logits:

$$\hat{G}_{t,i} = \frac{e^{r_{t,i}}}{\sum_{j \in S_t} e^{r_{t,j}}}$$

Biases are updated after each step by a simple rule:

$$b_i \leftarrow b_i - \gamma \cdot \operatorname{sign}(\text{load}_i - \bar{\text{load}})$$

$\gamma$ is a small step size (e.g. 0.001). Overloaded experts get a lower bias → fewer tokens next step.

Why this is better than aux loss
  • Aux loss directly competes with task loss — there is no free lunch; a larger $\alpha$ sacrifices model quality for balance.
  • Bias terms affect only routing decisions, not gradient updates. Task loss is uncontaminated.
  • The bias update is a simple sign-gradient step — essentially a per-expert EMA of overload signal.
DeepSeek-V3 numbers
  • 671B total parameters, 37B active per token.
  • 256 routed + 1 shared expert, top-8 active.
  • Trained on 14.8T tokens. No auxiliary balance loss used.
  • Expert load standard deviation < 3% across experts.
This approach is likely to become the standard. It eliminates an entire hyperparameter ($\alpha$) and its interaction with learning rate and model size.

XVIII · Router Init
Getting training started right

Router initialization and the first thousand steps determine whether specialization ever emerges.

1
Small weight init. Use $\sigma = 0.01$ for $W_r$. Large initial logits let one expert dominate immediately — the others get zero gradient for their FFN weights and never recover.
2
Temperature annealing. Divide logits by $\tau > 1$ early in training: $r_t / \tau$. Soft routing encourages exploration. Anneal $\tau \to 1$ over the first 5–10% of steps.
3
Jitter noise. Add $\epsilon \sim \mathcal{U}(0, 1/N)$ to logits before top-k. Breaks symmetry cheaply without a hyperparameter schedule.
4
Separate LR for router. Use 0.1× the expert learning rate for $W_r$. The router should adapt slowly relative to the experts it is directing. 
Router with jitter + temperaturedef route(self, x, training=False, tau=1.0):
    logits = x @ self.W_r.T        # [B, T, N]
    if training:
        noise = torch.rand_like(logits)
        logits = logits + noise / self.n
    logits = logits / tau
    gates_all = logits.softmax(-1)
    topv, topi = logits.topk(self.k, -1)
    gates = topv.softmax(-1)       # renorm over top-k
    return gates, topi, gates_all

Early collapse is permanent

If all tokens route to expert 0 in the first 500 steps, other experts receive zero gradient on their FFN weights. Increasing $\alpha$ after the fact does not recover specialization. Restart from scratch — or prevent it with the steps above.

Watch the per-expert token count histogram in the first 1000 steps. If any bar is 10× the others, intervene immediately.

XIX · Overflow
When an expert is full

Tokens beyond expert capacity must be dropped, rerouted, or padded — each has a cost.

Expert capacity formula

$$C = \left\lceil c_f \cdot \frac{T \cdot k}{N} \right\rceil$$

$c_f = 1.0$ allows zero slack. $c_f = 1.25$ gives 25% headroom at 25% extra memory. $c_f = 2.0$ guarantees near-zero drops but doubles expert buffer memory.

Strategy Overflow handling Trade-off
Drop Token passes through residual unchanged. Fast; gradient silent on dropped tokens.
Reroute Token goes to next-best expert (top-k+1). No info loss; can cascade overload.
Pad & mask Expert slot padded; output masked to zero. Fixed shapes; wastes GEMM compute.
Expert-choice Each expert picks exactly $C$ tokens — no overflow. Some tokens may never be picked.
Capacity factor intuition

With perfect balance, each expert sees $Tk/N$ tokens. Any imbalance causes drops at $c_f=1.0$.

Drop-on-overflow is the standard in Switch Transformer and most production systems. Rerouting is appealing conceptually but creates second-order load spikes that can be worse than the original imbalance.
Rule of thumb

Start with $c_f = 1.25$ during training and measure actual overflow rate. If it stays below 1%, reduce to 1.0 to save memory. Never use $c_f < 1.0$.

What healthy training looks like

Four curves tell you whether the router is learning, collapsing, or stalling.

Router entropy
Entropy should fall from random but not collapse to near-zero too fast.
Expert load histogram
Balanced routing keeps expert batches similar in size.
Overflow rate
Persistent overflow means capacity is too small or load balance is poor.
Aux balance loss
A balanced router still has nonzero aux loss — zero means uniform routing.
Failure signature

Entropy crashes within first 1k steps + load histogram spikes on 1–2 experts + overflow stays high.

Healthy signature

Entropy declines gradually over 10–50k steps while expert histogram stays uniformly distributed.

Common fix

Increase $\alpha$, add jitter noise, lower router LR, or raise $c_f$. Try one change at a time.

XXI · Variants
Design choices and their tradeoffs

Top-1, top-2, shared experts, expert-choice, and soft routing trade stability against cost.

Variant Main idea Strength Cost / Risk
Top-1 (Switch) Each token assigned to one expert. Cheapest dispatch; clearest specialization. Brittle — routing error is unrecoverable per token.
Top-2 (Mixtral) Each token uses two experts, outputs mixed. Redundancy; smoother gradients. 2× all-to-all traffic; 2× active FLOPs.
Shared + routed (DeepSeek) $K_s$ dense shared + $K_r$ sparse routed experts. Shared experts prevent knowledge silos. Partly re-introduces dense compute for shared path.
Expert-choice (Zhou 2022) Each expert selects its top-$C$ tokens. Perfect load balance; no aux loss needed. Tokens may be selected 0 or many times; normalisation complex.
Soft MoE (Google 2023) Continuous dispatch; all experts run on slot mixtures. Fully differentiable; zero dropped tokens. Not sparse — all-expert compute.
Loss-free bias (DeepSeek-V3) Per-expert bias in routing; updated by sign-gradient. No $\alpha$ hyperparameter; task loss uncontaminated. Requires careful step-size $\gamma$ tuning.
Choose top-1 when

You need max throughput and have robust balancing control. Good for inference-optimised deployments.

Choose top-2 when

Stability and quality matter more than inference cost. Industry default for large LMs.

Choose expert-choice when

You want guaranteed balance without an aux loss and can tolerate uneven per-token processing.

XXII · Inference
MoE at serving time — different problem than training

MoE inference is dominated by memory bandwidth and tiny expert batch sizes, not FLOPs.

Why inference is harder
  • At training with large batches, each expert processes many tokens per step — GEMMs are efficient.
  • At inference with batch size 1, each active expert runs a tiny matmul — memory-bound, not compute-bound.
  • All expert weights must still reside in memory (or be fetched from other devices) even though only 2/8 are active per token.
Arithmetic intensity at inference

$$\text{AI} = \frac{\text{FLOPs}}{\text{bytes loaded}} = \frac{2 \cdot d \cdot d_{ff}}{(d + d_{ff}) \cdot 2\,\text{B}} \ll \text{GPU ridge point}$$

For batch size 1, AI ≈ 1 FLOP/byte — far below the hardware ridge point (~50–200). Every expert call is pure bandwidth bottleneck.

Expert offloading

Keep inactive experts on CPU RAM or NVMe. Prefetch the next-predicted experts based on recent routing statistics.

Speculative routing

Predict which experts will be needed for the next token before committing, hiding transfer latency.

Expert caching

Maintain a fast cache of recently used experts in GPU VRAM. Routing locality means the same experts recur frequently.

MoE is not a free lunch at inference

Mixtral 8×7B has ~47B parameters but each forward pass loads ~12.9B worth of weights plus the inactive expert weights that must reside somewhere. Serving cost is dominated by total parameter memory, not active FLOPs.

Rule of thumb: MoE is great for training throughput at fixed compute. For latency-sensitive inference with small batch sizes, dense models of equal active-parameter count are often faster.

XXIII · Summary
The complete picture

MoE is a conditional compute architecture. Its value is real; its engineering cost is also real.

Core mechanism
  • Router scores tokens against $N$ experts with a linear map.
  • Top-k selection keeps $k \ll N$ gates non-zero.
  • Only selected experts execute; outputs are gate-weighted and summed.
  • Total params scale with $N$; active FLOPs scale with $k$.
Router landscape
  • Linear top-k — default, content-adaptive, collapse risk.
  • Noisy top-k — adds exploration noise, Shazeer 2017.
  • Expert-choice — experts pick tokens, perfect balance.
  • Soft MoE — differentiable, no sparsity.
  • Bias-based — loss-free balance, DeepSeek-V3.
Training essentials
  • Small router init ($\sigma=0.01$) + jitter noise.
  • Aux balance loss $\mathcal{L}_\text{balance}$ + z-loss $\mathcal{L}_z$.
  • Capacity factor $c_f \in [1.0, 2.0]$; monitor overflow rate.
  • Watch router entropy + load histogram every 1k steps.
SOTA landmarks

Switch-C (2021): 1.6T params, top-1. Mixtral 8×7B (2024): 47B total, 12.9B active, open weight. DeepSeek-V3 (2025): 671B total, 37B active, auxiliary-loss-free balance.

Bottom line

MoE is not a free lunch. It trades dense arithmetic for routing decisions, token shuffling, all-to-all communication, and systems complexity. When that trade is engineered well, model capacity can increase 4–8× faster than active per-token compute — enabling trillion-parameter models on practical hardware budgets.

Mental model: attention = communication layer (which tokens interact). MoE = conditional processing layer (which computation each token receives). They are complementary, not competing.
