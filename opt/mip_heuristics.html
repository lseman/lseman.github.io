§ 00

Why Do We Need Primal Heuristics?

🚨 Common Pitfall

Don't confuse primal heuristics with branch-and-cut! Primal heuristics find feasible solutions without proving optimality. They don't add cuts to the formulation. The goal is speed: find a good solution quickly, even if it's not guaranteed to be optimal.

A Mixed-Integer Program (MIP) extends linear programming by requiring some or all variables to take integer values. The general form is:

$$\begin{aligned} \min_{x} \quad & c^\top x \\ \text{s.t.} \quad & Ax \leq b \\ & x_j \in \mathbb{Z}, \quad j \in \mathcal{I} \subseteq \{1,\ldots,n\} \\ & \ell \leq x \leq u \end{aligned} \tag{MIP}$$

An Analogy: The Maze with Integer Rules

Think of a MIP as a maze where most of the space is continuous, but you must land on integer grid points at the end. The LP relaxation is like walking anywhere in the maze — you can visit any fractional point. But the true solution must be on the grid.

Branch-and-bound is like systematically exploring every corridor and room, checking each grid point one by one until you're certain you've found the best one. It's thorough, but slow.

Primal heuristics are like clever shortcuts: instead of checking every grid point, they use hints from the continuous walk (the LP solution) to quickly spot promising integer spots. They don't prove you've found the absolute best — but they often find good enough solutions very fast, which helps you prune the maze faster.

Integrality Gap — The Maze Width
The gap between the LP relaxation bound $z_{LP}$ and the best known integer solution $z^*$ is like the width of the maze. A small gap means the continuous walk is close to a grid point (easy shortcut). A large gap means you're far from any grid point (harder to find a good shortcut).

More formally, in a maximisation problem with incumbent value $z^{inc}$ and LP bound $z_{LP}(N)$ at node $N$, branch-and-bound prunes whenever $z_{LP}(N) \le z^{inc}$. A primal heuristic is therefore valuable not only because it finds a feasible point, but because it can improve $z^{inc}$ early and globally strengthen the pruning test across the search tree.

How a Heuristic Works: Four Simple Steps

Every primal heuristic follows a similar pattern — think of it as a recipe for finding integer solutions:

  • 1Look at the clues: the heuristic reads solver state, typically an LP solution, incumbent, reduced costs, or node context. Think of this as examining the continuous maze to find where grid points might be nearby.
  • 2Make a projection: it either projects toward integrality (like rounding) or restricts the search to a smaller neighborhood (like focusing on a specific corridor).
  • 3Generate candidates: integer assignments are produced by rounding, local moves, or a restricted sub-MIP. This is like trying different grid points near your promising area.
  • 4Check and keep: only candidates that satisfy the original MIP and improve the incumbent are retained. If a candidate violates constraints, it's discarded like stepping off the maze path.

Why Heuristics Matter — Three Key Benefits

Primal heuristics are algorithms that quickly find feasible integer solutions — often before the B&B tree fully closes the gap. Their key benefits:

⚡
Pruning Power
Analogy: Like finding a shortcut that reveals you're closer to the exit than you thought. A good incumbent tightens the bound, enabling aggressive pruning of the B&B tree — fewer rooms to search.
⏱️
Anytime Behaviour
Analogy: Like having a good-enough answer ready before you finish all your calculations. Even if B&B is interrupted, we have a feasible solution to report to the decision-maker.
🎯
Quality Focus
Analogy: Like a good GPS that learns your destination and takes increasingly better routes. Heuristics guide search toward promising regions, improving solution quality over time.

Taxonomy of Primal Heuristics

We study two families, mirroring the code structure:

Construction Heuristics

The Running Example

Throughout all interactive demos, we use the same 2-variable MIP. It is small enough to visualize geometrically, yet rich enough to expose the key behaviours of each heuristic.

$$\begin{aligned} \max \quad & 11x_1 + 6x_2 \\ \text{s.t.} \quad & 2x_1 + x_2 \leq 14 \\ & x_1 + 2x_2 \leq 14 \\ & x_1, x_2 \geq 0, \quad x_1, x_2 \in \mathbb{Z} \end{aligned} \tag{P}$$

The LP relaxation (dropping integrality) has its optimal vertex at the intersection of the two binding constraints:

$$2x_1^* + x_2^* = 14, \quad x_1^* + 2x_2^* = 14 \quad \Longrightarrow \quad x^* = \left(\tfrac{14}{3},\, \tfrac{14}{3}\right) \approx (4.67,\, 4.67)$$ $$z_{LP} = 11 \cdot \tfrac{14}{3} + 6 \cdot \tfrac{14}{3} = \tfrac{238}{3} \approx 79.33$$

Step-by-Step Derivation

  • 1Drop integrality to obtain a continuous LP relaxation.
  • 2At the LP optimum, both resource constraints are active, so solve the $2 \times 2$ linear system given by the equalities.
  • 3Evaluate the objective at the fractional point to obtain an upper bound for the integer maximisation problem.
  • 4Inspect nearby lattice points. This is the region that primal heuristics will exploit.

Since $x^*$ is fractional, we need heuristics to find an integer feasible point. The true integer optimum is:

Integer Optimal Solution
$x^\star = (5, 4)$ with objective $z^\star = 11 \cdot 5 + 6 \cdot 4 = 79$. The integrality gap is only $79.33 - 79 = 0.33$ ($\approx 0.4\%$), yet naive rounding fails to reach it.
Python Sketch — Enumerating the Tiny Running Example
best = None
for x1 in range(15):
    for x2 in range(15):
        if 2 * x1 + x2 <= 14 and x1 + 2 * x2 <= 14:
            obj = 11 * x1 + 6 * x2
            if best is None or obj > best[0]:
                best = (obj, (x1, x2))

print(best)  # (79, (5, 4))

Second Running Example — A Weak-Relaxation Binary Knapsack

The first example is deliberately friendly: the integrality gap is tiny and a carefully chosen rounding rule already succeeds. To understand why modern solvers still need stronger heuristics, it helps to contrast it with an instance whose LP relaxation is much weaker.

$$\begin{aligned} \max \quad & 10x_1 + 10x_2 + 10x_3 \\ \text{s.t.}\quad & 6x_1 + 6x_2 + 6x_3 \le 10,\\ & x_1,x_2,x_3 \in \{0,1\}. \end{aligned} \tag{K}$$

The integer problem can select at most one item, so the true optimum is $z^\star = 10$. The LP relaxation, however, can split capacity fractionally:

$$x^{LP} = \left(1,\; \tfrac{2}{3},\; 0\right) \quad \text{or any permutation, with} \quad z_{LP} = 10 + 10\cdot\tfrac{2}{3} = 16.67.$$
Why This Example Matters
The gap is now large: $16.67 - 10 = 6.67$, so the LP solution gives a misleadingly optimistic picture. Nearest rounding can easily produce $(1,1,0)$, which violates the knapsack constraint, while rounding down gives a feasible but low-information point. This is exactly the setting where feasibility pump, feasibility jump, and neighborhood sub-MIPs become substantially more valuable.

§ 02

Rounding Heuristic

What is Rounding? The Nearest Grid Point

The simplest idea: take the LP optimal solution $x^*$ and round each fractional integer variable to the nearest integer. If the result is feasible, we have a MIP solution.

Analogy: Imagine you're standing at coordinate $(4.67, 4.67)$ on a map, and you need to move to the nearest grid intersection. Rounding says "just step to the closest grid point" — you'd go to $(5, 5)$. Simple, but it doesn't know about the maze walls (constraints).

Scientifically, rounding is a projection from the continuous relaxation onto the integer lattice. Its cost is essentially linear in the number of integer variables, but closeness to the LP optimum does not imply feasibility with respect to the discrete constraint set.

$$\bar{x}_j = \begin{cases} \lfloor x_j^* \rceil & \text{if } j \in \mathcal{I} \text{ (round to nearest)} \\ x_j^* & \text{if } j \notin \mathcal{I} \end{cases}$$

Objective-Guided Rounding — Choosing the Best Grid Point

A smarter variant breaks ties using the objective coefficient $c_j$:

$$\bar{x}_j = \begin{cases} \lceil x_j^* \rceil & \text{if } x_j^* \text{ is fractional and rounding up improves } c^\top x \\ \lfloor x_j^* \rfloor & \text{otherwise} \end{cases}$$

Analogy: Imagine two grid points near your location — one is slightly closer, but the other is in a direction where you earn more points. A greedy approach picks the closer point; objective-guided rounding picks the point that gives more points.

In our maximisation example, the coefficient $c_1 = 11 > c_2 = 6$, so we prefer to round $x_1$ up. Objective-guided rounding of $(4.67, 4.67)$ first rounds $x_1 \to 5$ (higher coefficient), then checks constraints, giving $(5, 4)$ — the integer optimum!

Step-by-Step Mechanics

  • 1Solve the LP relaxation and identify the integer-constrained variables.
  • 2Apply a rounding rule: nearest, floor, ceiling, or an objective-guided variant.
  • 3Check all original MIP constraints, since rounding can break feasibility immediately.
  • 4If the point is infeasible, pass control to a repair or neighborhood heuristic.

Python Sketch — Objective-Guided Rounding
import math

def objective_guided_round(x_lp, c):
    rounded = []
    for value, coeff in zip(x_lp, c):
        frac = value - math.floor(value)
        if frac == 0:
            rounded.append(int(value))
        elif coeff >= 0:
            rounded.append(math.ceil(value))
        else:
            rounded.append(math.floor(value))
    return rounded

c = [11, 6]
print(objective_guided_round(x_lp, c))

§ 03

Diving Heuristics

Diving heuristics simulate a depth-first search (DFS) down a single branch of the branch-and-bound tree. Starting from an LP solution, they iteratively fix one (or a few) integer variable(s) to an integer value, re-solve the resulting LP relaxation (usually with dual simplex and warm-start), and repeat until all integer variables are fixed or a feasible MIP solution is found.

Key Idea
Good branching rules for the main B&B tree are often bad for diving — and vice versa. Diving prefers "safe" or "promising" fixings that quickly lead toward feasibility or good objective values.

Analogy: Think of diving as going down a single corridor in the maze until you hit a wall (infeasibility) or find an exit (feasible solution). Unlike branch-and-bound which explores many corridors simultaneously, diving focuses on one path at a time, making it fast but potentially missing the global optimum.

Unlike pure Rounding (no re-optimization), diving performs a sequence of LP re-solves. It is much cheaper than sub-MIP methods (RENS, RINS, Local Branching) but more powerful than one-shot rounding.

Common Diving Strategies — Different Ways to Choose Your Path

Fractional Diving
Fix the least fractional variable first (closest to an integer).
Analogy: Like fixing variables that are already almost on the grid — they're "stable" and less likely to cause problems.
Guided Diving
Fix variables toward a reference solution (usually the current incumbent).
Analogy: Like using a compass pointing toward a known good location — you're not just fixing any variable, you're fixing toward where you've already found something good. Especially powerful once any feasible solution exists.
Coefficient Diving
Prioritize variables with large objective coefficients (in maximisation, prefer fixing high-impact variables).
Analogy: Like fixing the variables that matter most for your score first — you lock in the "heavy hitters" early.
Vector Length / Lock Diving
Fix variables that appear in few constraints (low "locks") or have short column vectors first.
Analogy: Like fixing variables that have few connections to other constraints — they're "easy" to fix without causing chain reactions.
$$\text{At each dive step: } \quad \text{choose } j^* = \arg\min_{j \in \mathcal{I}_\text{frac}} \;\; \text{score}(j)$$ $$\text{then fix } x_{j^*} \leftarrow \lfloor x_{j^*}^{LP} \rceil \;\; \text{or guided value}, \quad \text{re-solve LP}$$

Step-by-Step Mechanics

  • 1Solve (or take) the current LP relaxation $x^{LP}$.
  • 2Score all fractional integer variables according to the chosen rule (fractionality, guidance, coefficient, …).
  • 3Fix the highest-priority variable to an integer value and tighten bounds.
  • 4Re-optimize the LP (warm-start). If infeasible, backtrack or abort the dive. Otherwise continue.
  • 5When all integer variables are fixed (or a depth limit is reached), check MIP feasibility. If feasible and improving, update incumbent.
Practical Notes
Most solvers limit dive depth (e.g., 20–100 steps) and allow limited backtracking (usually 1 level). Diving is often called at the root node and after new incumbents. It pairs very well with Fix-and-Propagate (replace LP re-solve with propagation at some steps).
Algorithm — Generic Diving Heuristic
1function DivingHeuristic(x_LP, problem P, scoring_rule, max_depth):
2   current_bounds ← copy of original bounds
3   for depth = 1 to max_depth:
4     x ← SolveLP(P, current_bounds)   // warm-start
5     if x infeasible: break or backtrack
6     j ← SelectVariableToFix(x, scoring_rule)

8       if IsMIPFeasible(x): return round(x)

11    current_bounds[j] ← {value, value}   // fix
12   return null   // dive failed

Feasibility Pump

What is the Feasibility Pump? Alternating Projections

When rounding produces an infeasible point, the Feasibility Pump (Fischetti et al., 2005) iterates between two projections until a feasible integer point is found:

Analogy: Imagine you're trying to reach a target on a wall (feasibility) while also trying to minimize distance from a starting point (LP solution). The pump alternates between: "move directly toward the target" (integer projection) and "move directly toward the starting point while staying on the wall" (LP projection). It's like two people pulling a rope from different directions — eventually they meet somewhere that satisfies both goals.

The key idea is to separate the two hard requirements of a MIP solution: integrality and constraint feasibility. The LP projection enforces the linear constraints, while the integer projection enforces integrality. Alternating these projections is not guaranteed to converge theoretically, but in practice it is one of the most effective root-node construction heuristics.

  • 1LP Projection: Solve an LP that finds the point in the LP feasible set $P$ closest (in $\ell_1$ norm) to the current rounded integer point $\bar{x}$.
  • 2Integer Projection: Round the LP solution $x$ to the nearest integer point $\bar{x}$ (component-wise).
  • 3If $\bar{x}$ is LP-feasible (i.e., $\bar{x} \in P$), it is MIP-feasible. Stop. Otherwise, repeat.
$$x^{k+1} = \underset{x \in P}{\arg\min}\; \sum_{j \in \mathcal{I}} \left| x_j - \bar{x}^k_j \right|$$ $$\bar{x}^{k+1} = \lfloor x^{k+1} \rceil \quad \text{(round to nearest integer)}$$

Step-by-Step Mechanics

  • 1Start from the LP solution $x^{LP}$ and round it to an integer vector $\bar{x}^0$.
  • 2Solve a distance-minimisation LP that pulls the continuous point back toward $\bar{x}^k$ while remaining LP-feasible.
  • 3Round again. If the rounded point satisfies the original MIP constraints, terminate.
  • 4If the same rounded point reappears, perturb the iterate to escape the cycle.

Cycle Detection & Perturbation

The pump can cycle (revisit the same rounded point). Detection is done via a signature hash of the candidate. Upon detecting a cycle, variables in violated constraints are flipped or perturbed:

$$\bar{x}_j \leftarrow 1 - \bar{x}_j \quad \text{(for binary variables)}$$

Two-Stage Approach

The implementation uses a two-stage strategy: first pump over binary variables only (easier to project), then extend to all integer variables. This mirrors the projection stages in the code: ProjectionStage::BinaryOnly then ProjectionStage::AllIntegers.

Linearised Projection Model

In an actual implementation, the $\ell_1$ projection is written as an LP by introducing auxiliary variables $d_j \ge 0$ that upper-bound the absolute deviation from the rounded target:

$$\begin{aligned} \min_{x,d}\quad & \sum_{j \in \mathcal I} d_j \\ \text{s.t.}\quad & Ax \le b,\\ & x_j - \bar x_j^k \le d_j \qquad (j \in \mathcal I),\\ & \bar x_j^k - x_j \le d_j \qquad (j \in \mathcal I),\\ & \ell \le x \le u. \end{aligned}$$

This is the main reason feasibility pump is attractive in practice: the repair step is still an LP, so solvers can use warm starts, basis information, presolve, and sparse linear algebra from the main branch-and-bound engine.

The repair LP then returns a feasible continuous point that is still close to that integer pattern. Advanced Pump Variants Used in Solvers

  • 1Objective blending: some variants minimise distance plus a small objective term so the repaired LP point is not only close to the rounded target but also directionally useful for the original objective.
  • 2Cycle hashes and perturbation: repeated rounded signatures trigger random flips, biased flips on violated rows, or restart logic.
  • 3Binary-first stages: for mixed binary/general-integer models, solvers often pump only over binaries first because that part of the integrality structure is most combinatorial.
  • 4Sub-MIP intensification: if the rounded point is promising but still infeasible, the solver may launch a tiny restricted MIP around it rather than continue pumping indefinitely.

import numpy as np

def round_nearest(x):
    return np.rint(x).astype(int)

def l1_distance(x, x_bar):
    return np.abs(x - x_bar).sum()

def pump_iteration(lp_solver, x_ref):
    x_bar = round_nearest(x_ref)
    x_next = lp_solver.closest_feasible_point(x_bar)  # abstract LP call
    return x_bar, x_next, l1_distance(x_next, x_bar)

Feasibility Jump

What is Feasibility Jump? Local Search in Integer Space

The Feasibility Jump is a Lagrangian-based local search that works directly in the integer space without solving LPs at each step. It is particularly effective when the LP relaxation is tight.

Analogy: Imagine you're standing on a grid point (an integer solution), and you want to find a better one nearby. Instead of checking every grid point (too slow), you use a "scorecard" that tells you how bad you are at violating rules and how good your score is. At each step, you take one small move that improves this scorecard — like stepping toward a better grid point while trying to stay within the maze walls.

In contrast to feasibility pump, feasibility jump does not alternate with a continuous projection. Instead, it defines a surrogate merit function whose minimisers are encouraged to reduce constraint violation first and improve the objective second. This makes the method attractive for large-scale MIPs where repeated LP solves would dominate runtime.

Lagrangian Scoring

Each candidate integer point $x \in \mathbb{Z}^n$ is scored by a weighted penalty for constraint violations plus a normalised objective term:

$$f_\lambda(x) = \underbrace{\sum_{i=1}^{m} \lambda_i \cdot v_i(x)}_{\text{weighted violation}} + \alpha \cdot \underbrace{\frac{c^\top x}{|z_{LP}|}}_{\text{normalised objective}}$$ $$v_i(x) = \max\!\left(0,\, a_i^\top x - b_i\right) \quad \text{(violation of constraint } i \text{)}$$

The multipliers $\lambda_i \geq 0$ are updated to emphasize violated constraints:

$$\lambda_i^{k+1} = \begin{cases} \min\!\left(10^6,\; \lambda_i^k \cdot \rho + \epsilon_i\right) & \text{if } v_i(x^k) > 0 \quad \text{(increase)}\\ \max\!\left(10^{-4},\; \lambda_i^k \cdot \delta\right) & \text{if } v_i(x^k) = 0 \quad \text{(decrease)} \end{cases}$$ \text{where } \rho \approx 1.05\text{–}1.2, \quad \delta \approx 0.995\text{–}0.98

Move Evaluation

At each iteration, for each integer variable $j$, we consider candidate new values:

$$\mathcal{M}_j = \begin{cases} \{1-x_j\} & \text{binary} \\ \{\lfloor x_j^{LP} \rceil,\; x_j+1,\; x_j-1,\; \text{guided}(x_j)\} & \text{general integer} \end{cases}$$

The score of a move $(j, v)$ is computed efficiently using column structure:

$$\Delta f(j, v) = \sum_{i : a_{ij} \neq 0} \lambda_i \left[v_i(x + (v - x_j)e_j) - v_i(x)\right]$$

Step-by-Step Mechanics

  • 1Construct an initial integer point, usually by rounding the LP solution.
  • 2Compute constraint violations and initialise Lagrangian multipliers.
  • 3Evaluate candidate local moves using sparse column information.
  • 4Take the best improving move, update multipliers, and diversify when the search stalls.

Worked Numerical Move Table

Suppose the current integer point is $(5,5)$, so the first constraint is violated by $1$ and the second is feasible. Let $\lambda=(2.0,1.0)$, $\alpha=0.1$, and $z_{LP}=79.33$. Consider three candidate moves:

Candidate Constraint Violations Penalty Term Objective Term Total Score
$(5,4)$ $(0,0)$ $0$ $0.1 \cdot 79/79.33 \approx 0.100$ 0.100
$(4,5)$ $(0,0)$ $0$ $0.1 \cdot 74/79.33 \approx 0.093$ 0.093

This example exposes the key modeling choice in feasibility jump: the objective term must be scaled so that reducing violation dominates while the point is still infeasible. Once multiple moves yield zero violation, the objective term becomes the tie-breaker. Solver implementations often switch more explicitly between a feasibility phase and an improvement phase for exactly this reason.

⚠️ Common Pitfalls

1. Lagrangian weights initialization: Starting with $\lambda_i = 0$ gives no penalty for violations. Use small positive values (e.g., 1.0 or 10.0) instead.

2. Objective scaling: If the objective term is too large, the heuristic focuses on improvement instead of feasibility. Always normalise by $|z_{LP}|$ or scale by a factor like $\alpha = 0.01$.

3. Move set too small: Only considering $\{x_j \pm 1\}$ for general integers may get stuck in local optima. Include guided moves toward the LP solution.

💡 Practical Tip

When to use Feasibility Jump: Large-scale binary MIPs where LP solves are expensive. The method excels at finding feasible solutions quickly without iterating on LPs.

Advanced Engineering Details

  • 1Incremental score updates: after changing one variable, only rows that contain that variable need to be rescored. This is why column-oriented sparse storage matters.
  • 2Plateau handling: if several moves have identical merit, solvers randomise or prefer moves with better objective impact to avoid deterministic stalling.
  • 3Multiplier schedules: aggressive increases in $\lambda_i$ force fast feasibility restoration; conservative schedules keep more room for objective improvement.
  • 4Restarts: after too many non-improving steps, the heuristic is often restarted from a new rounded LP point or from a perturbed incumbent.

def violation(A, b, x):
    return [max(0.0, sum(aij * xj for aij, xj in zip(row, x)) - bi)
            for row, bi in zip(A, b)]

def jump_score(A, b, c, x, lam, alpha, z_lp):
    viol = violation(A, b, x)
    return sum(li * vi for li, vi in zip(lam, viol)) + alpha * sum(ci * xi for ci, xi in zip(c, x)) / abs(z_lp)

def best_neighbor(neighbors, A, b, c, lam, alpha, z_lp):
    return min(neighbors, key=lambda x: jump_score(A, b, c, x, lam, alpha, z_lp))

Fix-and-Propagate Heuristic

Fix-and-Propagate is a lightweight and effective primal heuristic for mixed-integer linear programming (MIP). It combines an LP relaxation with constraint propagation: a subset of integer variables is fixed to integer values derived from the LP solution, after which bound propagation is used to tighten domains and recover a feasible integer solution.

Analogy: Think of it like solving a Sudoku puzzle. First, you make some obvious moves based on the clues (fix variables that are almost correct). Then, you propagate those choices — if a cell is fixed to 5, you update all other cells that interact with it. This chain reaction quickly narrows down what's possible for the remaining cells, often leading to a complete solution.

While the core idea has been used as folklore in MIP solvers for years, a strong modern variant with a repair phase is presented in:

Salvagnin, D., Roberti, R., & Fischetti, M. (2024). A fix-propagate-repair heuristic for mixed integer programming. Mathematical Programming Computation. DOI: 10.1007/s12532-024-00269-5

$$\text{Phase 1 (LP):} \quad \text{Solve LP relaxation to get } x^{LP}$$ $$\text{Phase 2 (Fix):} \quad \text{Fix } x_j = \round(x_j^{LP}) \text{ for a promising subset } J \subseteq \mathcal{I}$$ $$\text{Phase 3 (Propagate):} \quad \text{Tighten bounds } l_j, u_j \text{ via constraint propagation}$$ $$\text{Phase 4 (Complete):} \quad \text{Fix remaining variables and check feasibility (recurse if needed)}$$

Why It Works

The LP solution provides a high-quality fractional point that already respects all linear relaxations. Constraint propagation then exploits the logical structure of the constraints to rapidly force variables toward integer values or detect inconsistencies. This hybrid approach is often more effective and faster than pure LP rounding or standalone propagation:

  • LP rounding alone: frequently produces infeasible solutions.
  • Propagation alone: lacks good starting points and can be incomplete.
  • Combined: LP gives direction and quality, propagation enforces integrality cheaply and effectively.

Algorithm Sketch

  1. 1Solve the LP relaxation and obtain the fractional solution x^{LP}.
  2. 2Select and fix a subset of integer variables to their nearest integer values (usually those closest to integrality).
  3. 3Perform bound propagation (domain reduction) on the remaining variables using all constraints.
  4. 4If all integer variables become fixed and the solution is feasible, return it. Otherwise, fix remaining fractional variables by rounding and recurse or terminate.
Practical note
Fix-and-Propagate performs best on small to medium MIPs (typically fewer than 100–200 integer variables) where propagation remains fast. For larger instances it is often used as a quick diving heuristic inside branch-and-cut or combined with a repair phase (Salvagnin et al., 2024). It is computationally cheap compared to sub-MIP solves used in RINS or Feasibility Pump.

RENS — Relaxation Enforced Neighborhood Search

What is RENS? Fixing What's Already Correct

RENS (Berthold, 2014) fixes variables that are already integer in the LP relaxation, and solves a sub-MIP over the remaining fractional variables. The key insight: the LP relaxation itself defines the neighborhood.

Analogy: Imagine you're building a house and some walls are already perfectly vertical (integral variables). Instead of checking all walls, you only need to work on the crooked ones (fractional variables). RENS says "trust the LP — if a variable is already integer there, lock it in place and only explore where the LP is still uncertain."

The scientific rationale is that variables already integral in the LP optimum are often structurally stable. By fixing them, we preserve the LP solution's qualitative pattern while leaving freedom only where integrality is unresolved. In effect, RENS trades one hard MIP for a much smaller one.

$$\text{Fix: } x_j = x_j^{LP} \quad \text{if } x_j^{LP} \in \mathbb{Z} \quad (j \in \mathcal{I})$$ $$\text{Bound: } \lfloor x_j^{LP} \rfloor \leq x_j \leq \lceil x_j^{LP} \rceil \quad \text{if } x_j^{LP} \notin \mathbb{Z} \quad (j \in \mathcal{I})$$

The resulting sub-MIP has fewer free integer variables. If the fixing rate is high enough (controlled by rens_fix_ratio), the sub-MIP can be solved quickly.

Adaptive Fixing

When the initial sub-MIP fails (infeasible or no improvement), RENS tries several fixing ratios: among the fractional variables, those closest to integer (smallest $|x_j^{LP} - \lfloor x_j^{LP} \rceil|$) are fixed first.

$$\text{priority}(j) = \left|x_j^{LP} - \lfloor x_j^{LP} \rceil\right| + \epsilon \cdot |c_j| \quad \text{(smaller = fix sooner)}$$

Step-by-Step Mechanics

  • 1Inspect the LP solution and classify each integer variable as integral or fractional.
  • 2Fix integral LP variables exactly and bound fractional ones to their nearest integer interval.
  • 3Optionally fix additional near-integer variables according to a ranking rule.
  • 4Solve the restricted sub-MIP under a time budget and keep any improving feasible solution.

RINS — Relaxation Induced Neighborhood Search

What is RINS? Combining LP and Incumbent Wisdom

RINS (Danna et al., 2005) combines the LP relaxation and an existing incumbent solution. Variables where both agree are fixed; the rest define the neighborhood.

Analogy: Imagine you have two maps — one from a surveyor (LP relaxation) and one from a traveler's experience (incumbent). RINS says: "Where both maps agree, trust them — those locations are definitely good. Where they disagree, explore further." It's like combining the theoretical optimum with practical knowledge to find even better paths.

Compared with RENS, RINS injects additional information from the incumbent. Agreement between the LP solution and the incumbent acts as a stability signal: those variables are likely part of a good local structure. The resulting neighborhood is often smaller and better targeted than the one used by RENS.

$$\text{Fix: } x_j = x_j^{inc} \quad \text{if } \left|x_j^{LP} - \lfloor x_j^{inc} \rceil\right| \leq \tau \quad (j \in \mathcal{I})$$

where $\tau$ is the agreement tolerance rins_tolerance. The intuition: if the LP and the incumbent "agree" on a variable's integer value, fixing it is likely safe.

Fixing Priority for Disagreeing Variables

For variables where LP and incumbent disagree, RINS ranks them by:

$$\text{score}(j) = \left|x_j^{LP} - x_j^{inc}\right| - \beta \cdot |\bar{c}_j| - \gamma \cdot \left|x_j^{LP} - \lfloor x_j^{LP} \rceil\right|$$

where $\bar{c}_j$ is the reduced cost. Variables with small score (i.e., small LP–incumbent disagreement, large reduced cost, small fractionality) are fixed first.

Step-by-Step Mechanics

  • 1Read the current LP solution and incumbent at a node of branch-and-bound.
  • 2Fix every integer variable on which the two solutions agree up to tolerance.
  • 3Rank the remaining variables by disagreement and reduced-cost information.
  • 4Solve one or more restricted sub-MIPs and keep any improving incumbent.

DINS — Distance Induced Neighborhood Search

DINS (Distance Induced Neighborhood Search, Robert Waniek) combines the ideas of RINS (Relaxation Induced Neighborhood Search) and Local Branching. It defines the neighborhood by introducing a distance function between the incumbent solution and the optimum of the LP relaxation. When applied during a branch-and-bound search, it further takes into account how variables change their values at different nodes of the tree.

What is DINS? The Distance-Based Neighborhood

Analogy: Think of DINS as a "smart" version of RINS. While RINS just looks at whether LP and incumbent agree on a variable's value, DINS measures how far apart they are. It's like not just checking if two friends agree on a restaurant, but also considering how much they disagree (one wants Italian, the other wants Chinese).

DINS uses a distance metric to define which variables should be fixed and which should be explored. Variables with small distance (high agreement) are fixed; variables with large distance are freed for exploration.

Distance Function

DINS defines a distance measure between the LP solution and the incumbent:

$$\delta_j = \left| x_j^{LP} - x_j^{inc} \right| \quad \text{for binary variables}$$ $$\delta_j = \left| x_j^{LP} - \lfloor x_j^{inc} \rceil \right| \quad \text{for general integers}$$

Mental model: The distance $\delta_j$ represents how much the LP solution disagrees with the incumbent on variable $j$. Small distance = good agreement (fix it). Large distance = big disagreement (explore it).

Dynamic Distance Tracking

Unlike RINS, DINS goes one step further by tracking how the LP solution changes as we move through the branch-and-bound tree. This is particularly useful for understanding variable behavior at different nodes:

$$\Delta_j = \left| \delta_j^{node} - \delta_j^{parent} \right| \quad \text{(distance change from parent)}$$

Why this matters: Variables that change their LP value significantly between parent and child nodes may be "unstable" and should be explored more aggressively. DINS uses this information to adaptively choose which variables to fix and which to free.

Neighborhood Definition

DINS defines a neighborhood $\mathcal{N}$ around the incumbent by fixing variables with distance below a threshold $\tau$ and freeing the rest:

$$\text{Fix: } x_j = x_j^{inc} \quad \text{if } \delta_j \leq \tau \quad (j \in \mathcal{I})$$ $$\text{Free: } \lfloor x_j^{inc} \rfloor \leq x_j \leq \lceil x_j^{inc} \rceil \quad \text{if } \delta_j > \tau$$

The threshold $\tau$ can be chosen as a fraction of the maximum distance or based on the distribution of distances across all variables.

Step-by-Step Mechanics

  • 1Compute the LP solution at the current node of branch-and-bound.
  • 2Calculate the distance $\delta_j$ for each integer variable between the LP solution and the incumbent.
  • 3Optionally compute $\Delta_j$ (distance change from parent node) for dynamic adaptation.
  • 4Select a threshold $\tau$ and fix variables with $\delta_j \leq \tau$ to their incumbent values.
  • 5Bound the remaining variables to their integer neighborhood and solve the restricted sub-MIP.
  • 6If an improving solution is found, update the incumbent and continue; otherwise, return to branch-and-bound.
Key Innovation
DINS adds dynamic distance tracking to the RINS framework. By monitoring how the LP solution changes across nodes, DINS can adaptively adjust the neighborhood size and composition. This makes it more responsive to the problem structure than static RINS.

Why DINS is Powerful

📏 More Granular Selection
Unlike RINS which uses a binary agreement test, DINS uses a continuous distance measure. This allows more nuanced decisions about which variables to fix.
🔄 Adaptive Learning
By tracking distance changes across nodes, DINS learns about variable behavior and can adapt its neighborhood definition dynamically.

Local Search Heuristic

What is Local Search? The Sliding Window

The Local Search heuristic explores a small neighborhood around the current incumbent by freeing a sliding window of integer variables and fixing all others:

$$\mathcal{N}(x^{inc}, W) = \left\{ x \;\middle|\; x_j = x_j^{inc} \text{ for } j \notin W, \; x_j \text{ free for } j \in W \right\}$$

Analogy: Imagine you're walking along a path and you want to try a slightly different route. Instead of changing your entire path (too expensive), you pick a small section (the "window") and explore alternatives there, while keeping the rest of your path intact. It's like trying a detour for just a few blocks while returning to your main route afterward.

where $W \subset \mathcal{I}$ is a window of $|W|$ variables selected by disagreement with the LP relaxation:

$$W = \operatorname*{argmax}_{|S|=w} \sum_{j \in S} \left|x_j^{LP} - \lfloor x_j^{inc} \rceil\right|$$

Mental model: The window focuses on variables that differ most between the LP relaxation and the incumbent — these are the variables most likely to lead to improvement. Variables that differ most between the LP relaxation and the incumbent are the most promising to "unfix" — the LP hints at a better direction. The window slides across iterations, systematically exploring different parts of the neighborhood.

Step-by-Step Mechanics

  • 1Measure disagreement between the LP solution and the incumbent for every integer variable.
  • 2Select a window $W$ of the most promising variables and free only those variables.
  • 3Fix all remaining variables to the incumbent values and solve the resulting sub-MIP.
  • 4Shift the window and repeat, creating a sequence of local neighborhoods with controlled size.

1. Window size selection: Too small a window limits exploration (local optimum risk). Too large a window makes the sub-MIP expensive and loses the neighborhood benefit. Typical values: w = 10-20 for binary MIPs.

2. Ranking metric: Using only disagreement magnitude ignores reduced costs. Variables with large reduced costs may be important for feasibility. Combine both metrics: score(j) = disagreement + β·|reduced_cost|.

3. No iteration limit: Local search can run indefinitely if no improvement is detected. Always set a time or iteration budget to prevent excessive runtime.

💡 Practical Tip

When to use Local Search: When you have a good incumbent and want to refine it. Local search excels on binary MIPs with moderate size (n < 500) and works well as a post-processing step after other heuristics.

Local Branching

What is Local Branching? The Hamming Ball

Analogy: Imagine your incumbent solution is the center of a circle on a binary map. Local Branching defines a neighborhood as all binary points inside a circle of radius $k$ — these are points that differ from your incumbent by at most $k$ bits. It's like exploring a "Hamming ball" around your current solution.

Local Branching (Fischetti & Lodi, 2003) defines the neighborhood via a Hamming-distance constraint on the binary variables. Given an incumbent $\bar{x}$ with $k$ binary variables, the $k$-opt neighborhood is:

$$\sum_{j \in \mathcal{B}: \bar{x}_j = 1} (1 - x_j) + \sum_{j \in \mathcal{B}: \bar{x}_j = 0} x_j \leq k$$

This elegant constraint says: the new solution may flip at most $k$ binary variables from the incumbent. The MIP with this added cut is the local branching sub-problem.

Why this is powerful: By expressing the neighborhood as a linear inequality, a general MIP solver can explore it using all its machinery (cuts, presolve, propagation). It's a meta-heuristic implemented through exact methods — the best of both worlds.

Improving Cut

To ensure the solution strictly improves the incumbent objective $z^{inc}$, an additional cutting plane is added:

$$c^\top x \geq z^{inc} + \varepsilon \quad \text{(for maximisation)}$$

Agreeing Integer Variables

For general integer (non-binary) variables where the LP and incumbent agree, they are pre-fixed to reduce the sub-MIP size:

$$x_j = x_j^{inc} \quad \text{if } \left|x_j^{LP} - x_j^{inc}\right| \leq \tau_{lb}$$

Step-by-Step Mechanics

  • 1Take the incumbent binary vector as the center of the neighborhood.
  • 2Add a Hamming-ball constraint that allows at most $k$ binary changes.
  • 3Optionally add an improving cut so the solver searches only for strictly better incumbents.
  • 4Solve the sub-MIP under a limited budget; if no improvement is found, enlarge or move the neighborhood.

Large Neighborhood Search (LNS)

What is LNS? The Destroy-and-Repair Framework

Large Neighborhood Search (LNS) is one of the most important modern ideas in MIP heuristics. Instead of searching the entire combinatorial space again, LNS starts from a known feasible solution and explores a restricted neighborhood around it.

Analogy: Think of LNS like editing a document. Instead of rewriting the entire document (too slow), you select some paragraphs to delete and rewrite them (destroy), then use your word processor to fill in the blanks with better content (repair). The key insight: you keep what's already good, and only change what might be improvable.

The core principle is simple: keep part of the incumbent fixed, free another part, and then solve the resulting sub-MIP. This creates a neighborhood that is much larger than a tiny one-variable local search move, but still much cheaper than solving the full original model.

Unifying View
LNS is a broad framework. Many famous MIP heuristics can be interpreted as special cases of LNS: RINS fixes variables where incumbent and LP agree, Local Branching uses a Hamming-ball neighborhood, and Crossover fixes variables where good parents agree.
$$\text{Given incumbent } x^{inc}, \text{ choose a free set } F \subseteq \{1,\dots,n\}$$ $$x_j = x^{inc}_j \qquad \forall j \notin F$$ $$\text{solve the restricted sub-MIP over the free variables in } F$$

Destroy-and-Repair Interpretation — The Core Loop

In practice, LNS is often described as a destroy-and-repair heuristic:

  • 1Destroy: select some variables and release them from the incumbent assignment. Analogy: Like erasing part of your solution to explore alternatives.
  • 2Repair: solve the restricted sub-MIP to rebuild those variables in a better way. Analogy: Like filling in the erased part with something improved.
  • 3Accept: keep the repaired solution if it is feasible and improves the incumbent. Analogy: Like keeping the edit if it makes things better, otherwise undoing it.

This view is especially important pedagogically because it unifies many apparently different heuristics under the same logic: search a large but structured neighborhood around a good solution.

Why LNS is Powerful — Two Key Advantages

Much Larger Than Local Search
Analogy: A single-variable move is like taking one step. LNS frees many variables at once, so the solver can perform coordinated changes (like rearranging an entire room) that a purely local search would miss.
Single-variable moves explore tiny neighborhoods; LNS explores rooms full of possibilities.
Much Cheaper Than Full Re-optimization
Analogy: Rewriting your entire essay from scratch is expensive. LNS is like editing only the parts that need work — you save time because most of your solution is already good.
Because most variables remain fixed, the sub-MIP is far smaller than the original problem and can often be solved quickly.

Neighborhood Design Choices

The practical strength of LNS depends on which variables are freed. Several strategies are common:

Destroy Rule Main Idea Typical Use
Random destroy Free a random subset of variables Diversification, simplicity
Fractional-guided Free variables with high LP fractionality or disagreement LP-informed repair
Constraint-based Free variables involved in violated / tight rows Repair difficult structures
Block / domain-based Free one structural block at a time Scheduling, routing, assignment models

Scientific Perspective

From an optimization viewpoint, LNS defines a family of restricted feasible regions \(\mathcal{N}(x^{inc})\) around the incumbent. The heuristic then solves

$$\min \; c^\top x \qquad \text{subject to} \qquad x \in \mathcal{N}(x^{inc}) \cap X,$$

where \(X\) is the feasible set of the original MIP. The neighborhood is large enough to admit substantial improvement, but structured enough that solving inside it remains computationally affordable.

Worked Binary Example

Suppose the incumbent is the binary vector

$$x^{inc} = (1,0,1,1,0,0,1,0).$$

If an LNS heuristic frees positions \(2,4,7\), then the other variables remain fixed:

$$x_1=1,\;x_3=1,\;x_5=0,\;x_6=0,\;x_8=0,$$ $$x_2,\;x_4,\;x_7 \text{ are free and re-optimized by the sub-MIP.}$$

The search is therefore no longer over all \(2^8 = 256\) assignments, but only over the \(2^3 = 8\) assignments in the chosen neighborhood.

Adaptive Large Neighborhood Search (ALNS)

What is ALNS? Learning Which Tools Work Best

Adaptive Large Neighborhood Search (ALNS) extends LNS by not using a single fixed destroy-and-repair strategy. Instead, it maintains a portfolio of operators and adapts over time, rewarding operators that recently produced good improvements.

Analogy: Imagine you're a chef trying to create a perfect dish. You have various tools — a knife, a blender, a mortar and pestle. Standard LNS picks one tool (say, the knife) and keeps using it. ALNS is like a chef who tries different tools and learns which ones work best for different ingredients. If chopping vegetables works well today, the chef uses the knife more often tomorrow.

This is the natural next step after standard LNS. In ordinary LNS, the user chooses one destroy rule (random, block, fractional-guided, etc.). In ALNS, the heuristic itself learns which operators are working well on the current instance and allocates more calls to them.

Key Idea
ALNS treats destroy and repair mechanisms as competing operators. After each iteration, operator scores are updated according to outcome quality, and future selections are biased toward high-performing operators.
$$\text{At iteration } t,\quad \text{choose } d \in \mathcal{D},\; r \in \mathcal{R}$$ $$x^{cand} \leftarrow \text{Repair}_r\!\left(\text{Destroy}_d(x^{inc})\right)$$ $$\text{update operator scores based on feasibility and improvement}$$

Why ALNS is Useful

Adapts to Instance Structure
Some instances respond better to block destroy, others to LP-guided or random diversification. ALNS automatically shifts weight toward the operators that are currently effective.
Balances Intensification and Diversification
Strong operators are exploited more often, but weaker operators are still sampled occasionally so the search does not become too myopic.

Canonical ALNS Loop

  • 1Select a destroy operator and a repair operator using weighted randomization.
  • 2Apply them to produce a candidate solution inside a large neighborhood.
  • 3Accept or reject the candidate according to a search rule, such as improving-only or simulated annealing style acceptance.
  • 4Update scores of the selected operators according to the observed outcome.

Operator Scoring

A common pedagogical scoring rule is:

Outcome Reward Meaning
Global best high The operator pair produced the best solution seen so far.
Improving incumbent medium The candidate is accepted and improves the current incumbent.
Accepted but not improving small The move helps exploration but does not improve the best value.
Rejected / infeasible none No immediate contribution.
$$w_k^{t+1} = (1-\eta)\, w_k^t + \eta \cdot \text{reward}_k^t,$$

where \(w_k\) is the weight of operator \(k\) and \(\eta \in (0,1)\) is a reaction factor. Large \(\eta\) makes the method react quickly; small \(\eta\) makes it more stable.

Crossover & Mutation Heuristics

Analogy: Think of Crossover and Mutation as the two fundamental operations from genetic algorithms, adapted for MIP. Crossover is like two parents having a child that inherits their best traits. Mutation is like introducing a small random change to create diversity.

While RINS and Local Search work with a single incumbent and the LP solution, Crossover and Mutation explicitly exploit multiple known feasible solutions. These are classic ideas from genetic algorithms, adapted to MIP solvers for fast large-neighborhood improvement.

Crossover
Combine two (or more) good incumbents by fixing variables on which they agree. The resulting sub-MIP explores a promising “child” solution.
Mutation
Take one incumbent and randomly (or greedily) change a small number of variables, then repair via LP or propagation.

Crossover Mechanics

Given two feasible solutions $x^A$ and $x^B$, we create a new subproblem by fixing every integer variable where they agree:

$$x_j = x^A_j = x^B_j \quad \text{if } x^A_j = x^B_j \quad (j \in \mathcal{I})$$ $$\text{Free otherwise (or bound to } [\min(x^A_j,x^B_j), \max(x^A_j,x^B_j)] \text{ for general integers)}$$

The solver then solves this restricted sub-MIP (usually with a time/node limit). Because the two parents are good, the region where they agree is often high-quality.

Mutation Mechanics

Mutation is a lightweight perturbation of a single incumbent:

  • 1Select a small set of integer variables (1–5% or fixed window).
  • 2Flip, shift, or randomize their values (One-Opt, Two-Opt, or random walk).
  • 3Repair the resulting (possibly infeasible) point using LP projection, propagation, or a short Feasibility Jump.
  • 4If the new solution is feasible and improves the incumbent, accept it.
Why They Work Well
Crossover exploits common structure between good solutions.
Mutation adds diversity and helps escape local optima.
Both are very cheap when combined with warm-start LP or propagation.

Advanced Topics: Mathematical Foundations & GPU Acceleration

Mathematical Foundations — Convergence Analysis

While primal heuristics are not exact methods, they can be analyzed using tools from optimization theory. Consider the Feasibility Pump as a special case of an alternating projection algorithm.

Feasibility Pump as Alternating Projections

The Feasibility Pump alternates between two sets: the LP feasible set $P$ and the integer lattice $\mathbb{Z}^n$. Mathematically, we seek a point in the intersection $P \cap \mathbb{Z}^n$. The heuristic performs:

$$x^{k+1} = \Pi_P(x^k), \quad \bar{x}^{k+1} = \Pi_{\mathbb{Z}^n}(x^{k+1})$$ $$\text{where } \Pi_S(y) = \arg\min_{z \in S} \|z - y\|$$

Theoretical Result: The alternating projection algorithm converges to the closest point in $P \cap \mathbb{Z}^n$ under certain conditions (von Neumann, 1935 for convex sets; Feasibility Pump does not guarantee convergence in finite time).

RINS as Restricted Subproblem Optimization

RINS can be viewed as solving a sequence of restricted MIPs. Let $x^{inc}$ be the incumbent and define the neighborhood $N(x^{inc}, \tau)$:

$$N(x^{inc}, \tau) = \{ x \in X : x_j = x_j^{inc} \text{ if } |x_j^{LP} - x_j^{inc}| \leq \tau \}$$

The RINS heuristic solves:

$$\min \{ c^\top x : x \in N(x^{inc}, \tau) \}$$

Complexity Analysis: If the LP relaxation is tight (small integrality gap), RINS typically fixes a large fraction of variables, reducing the subproblem size by a factor proportional to the fixing rate.

DINS — Distance-Based Neighborhood Theory

DINS defines a continuous distance function $\delta: \mathbb{R}^n \to \mathbb{R}^n$. The neighborhood is defined by a threshold $\tau$:

$$N_{DINS}(x^{inc}, x^{LP}, \tau) = \{ x \in X : \delta_j(x) \leq \tau \implies x_j = x_j^{inc} \}$$

Adaptive Threshold Selection: Let $D = \max_j \delta_j$. Common choices for $\tau$:

$$\tau_{fixed} = \alpha \cdot D, \quad \alpha \in \{0.3, 0.5, 0.7\}$$ $$\tau_{percentile} = \text{percentile}_{p}(\{\delta_j\})$$ $$\tau_{adaptive} = \tau_0 \cdot \exp(-\beta \cdot \text{improvement\_rate})$$

GPU Acceleration of Primal Heuristics

Why GPU? Many primal heuristics involve operations that are highly parallelizable: distance computations, vector operations, and independent subproblem evaluations. GPUs provide massive parallelism for these workloads.

Parallelizable Component: Each dive step can be assigned to a GPU thread.
Implementation: Use CUDA streams for independent LP solves with warm-start.
Speedup: 5-10x speedup reported on large MIPs (n > 1000 variables).
⚡ Feasibility Pump
Parallelizable Component: Distance minimization in the LP projection is highly parallel.
Implementation: Use CUDA for sparse matrix operations and vector norms.
Speedup: 3-5x on binary MIPs.
🔍 RENS/RINS
Parallelizable Component: Multiple sub-MIP solves with different fix ratios.
Implementation: Launch kernel per subproblem; use shared memory for LP basis.
Speedup: 4-8x with 8-32 subproblems in parallel.
📊 Local Search
Parallelizable Component: Sliding windows can be evaluated in parallel.
Implementation: Each window = one thread; reduce to best solution.
Speedup: 6-12x with window sizes of 10-50 variables.

GPU Implementation Architecture

CUDA Architecture for DINS

Host (CPU): Manages branch-and-bound tree, incumbent updates, parameter tuning.
Device (GPU): Executes parallel heuristic kernels.
Data Flow:

  1. Transfer LP solution $x^{LP}$ and incumbent $x^{inc}$ to GPU (memory copy).
  2. Compute distances $\delta_j$ for all $n$ variables (parallel kernel).
  3. Select neighborhood based on $\tau$ (parallel reduction).
  4. Launch sub-MIP solver kernel (one per multiprocessor).
  5. Collect improving solutions, transfer back to host.

Code Example — CUDA Kernel for Distance Computation

CUDA Kernel — Compute DINS Distances
#include <cuda_runtime.h>
#include <device_launch_parameters.h>

__global__ void compute_distances_kernel(
    const float *x_lp,
    const float *x_inc,
    const int *var_types,  // 0=binary, 1=general integer
    float *distances,
    int n) {

    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx >= n) return;

    float lp_val = x_lp[idx];
    float inc_val = x_inc[idx];
    int var_type = var_types[idx];

    float dist;
    if (var_type == 0) { // Binary
        dist = fabsf(lp_val - inc_val);
    } else { // General integer
        dist = fabsf(lp_val - roundf(inc_val));
    }

    distances[idx] = dist;
}

void launch_dins_kernel(cudaStream_t stream) {
    int n = problem_n_variables;
    int threads_per_block = BLOCK_SIZE;
    int num_blocks = (n + threads_per_block - 1) / threads_per_block;

    cudaEvent_t start, stop;
    cudaEventCreate(&start);
    cudaEventCreate(&stop);

    // Record start time
    cudaEventRecord(start, stream);

    // Launch kernel
    compute_distances_kernel<<>>(
        d_x_lp, d_x_inc, d_var_types, d_distances, n);

    // Record end time

    if (err != cudaSuccess) {
        fprintf(stderr, "CUDA error: %s\n", cudaGetErrorString(err));
    }

    // Compute elapsed time
    cudaEventSynchronize(stop);
    float elapsed_ms = 0;  // (compute using cudaEventElapsedTime)
}

Practical Notes

Data Transfer Overhead: The biggest bottleneck is transferring data between CPU and GPU. Mitigation strategies:

  • Keep data on GPU for multiple heuristic calls (avoid repeated transfers).
  • Use pinned memory for faster host-to-device transfers.
  • Batch multiple LP solves in one transfer.

Memory Bandwidth: GPUs have high memory bandwidth (A100: 2 TB/s), making them ideal for matrix-vector operations common in LP solves.

Hybrid CPU-GPU: Many solvers use a hybrid approach: CPU for branch-and-bound orchestration, GPU for heuristic parallelism.

OpenMP vs GPU — When to Use What?

Use OpenMP When:
  • Problem size is moderate (n < 1000).
  • Memory bandwidth is not the bottleneck.
  • Code complexity is a concern.
  • Single-node deployment.
Use GPU When:
  • Problem size is large (n > 1000).
  • Many independent subproblems can be launched.
  • Matrix operations dominate runtime.
  • Cluster with GPU nodes available.

PyTorch/CPU Implementation — Modern Alternative

PyTorch provides an alternative to CUDA for GPU acceleration. The interface is more Pythonic and integrates well with existing Python MIP solvers.

PyTorch Implementation — DINS Distances
import torch
import torch.nn.functional as F

def compute_dins_distances_torch(x_lp, x_inc, var_types, tau=None):
    """Compute DINS distances using PyTorch (CPU or GPU)."""

    # Convert to tensors (automatically on GPU if tensors are on GPU)
    x_lp = torch.tensor(x_lp, device=x_lp.device)
    x_inc = torch.tensor(x_inc, device=x_inc.device)
    var_types = torch.tensor(var_types, device=var_types.device)

    # Compute distances in parallel
    if var_types.eq(0).all(): # All binary
        distances = F.abs(x_lp - x_inc)
    else:
        # Mixed: binary and general integer
        binary_mask = var_types.eq(0).float()
        general_mask = var_types.eq(1).float()
        binary_distances = F.abs(x_lp - x_inc) * binary_mask
        # For general integers: distance to nearest integer
        x_inc_rounded = torch.round(x_inc)
        general_distances = F.abs(x_lp - x_inc_rounded) * general_mask
        distances = binary_distances + general_distances

    # Compute threshold-based fixing
    if tau is None:
        tau = distances.max() * 0.5  # Default: 50th percentile

    fixed_mask = distances <= tau
    fixed_vars = x_inc * fixed_mask
    # Free variables: bound to [round(x_inc_j), ceil(x_inc_j)]
    free_lo = torch.floor(x_inc) * (~fixed_mask)
    free_hi = torch.ceil(x_inc) * (~fixed_mask)

    return distances, fixed_vars, free_lo, free_hi
